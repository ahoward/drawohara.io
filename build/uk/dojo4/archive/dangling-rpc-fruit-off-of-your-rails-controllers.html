При роботі з будь-яким додатком Rails, крім "hello-world", неминуче виникає потреба в невеликих AJAX-допоміжниках для спілкування з сервером, щоб виконувати клієнтську логіку. Я не маю на увазі API як таке, а радше ад-хок утиліти, такі як ці:

markdown.change(function(){
  generate_html_on_the_server( markdown.val(), function(html){
  
    preview.html(html);
  
  });
});

або

reservation.change(function(){
  check_availibity_on_the_server( reservation.val(), function(data){
  
    if(!data['available']){
     
     alert('цей час недоступний - виберіть інший!');
     
    };
  })
});

та інше.

Ці типи функцій не є частиною публічного API сервер-сервер, вони просто співпрацюють JS/бекенд ендпоінти, які потрібні для роботи функції перегляду.

Більшість додатків Rails накопичать багато таких функцій, і виникає питання:

«Куди їх помістити?»

У більшості команд буде три або чотири розробника, які будуть називати та організовувати ці функції по-різному, забезпечуючи, щоб кодова база перетворилася на безладну спагеті-систему за короткий час.

У @dojo4 ми абстрагували це з невеликим RPC шаблоном дизайну: спочатку ми маємо невеликий контролер DSL, який включений у наш ApplicationController і дозволяє декларативне визначення 'rpc' JS допоміжників на основі контролера.

class ApplicationController < ActionController::Base
  include(RPC)
end

Ознайомтесь з реалізацією тут: https://gist.github.com/ahoward/7320900

Цей DSL просто визначив одну дію на контролері, яка мультиплексує, який метод використовувати на основі параметрів, і легкий спосіб їх визначити. Він очікує, що всі дії RPC повернуть хеш і повернуть JSON.

Його реалізація зводиться до

  def rpc
    which = params['method']
    
    action = @rpc[which]
    
    result = action.call(params)
    
    render :json => result.to_json
  end

Його використання має бути очевидним з коду

class GeoLocationController < ApplicationController
  rpc(:geo_location) do |params|
    geo_location = GeoLocation.geo_locate( params['address'] )
    geo_location.attributes
  end
  
  rpc(:lat_lng) do |params|
   geo_location = GeoLocation.geo_locate( params['address'] )
   { 'lat' => geo_location.lat, 'lng' => geo_location.lng }
  end
end

Використання з JS потребує двох речей: маршруту та невеликого класу JS. Спочатку маршрут:

  match ':controller#rpc', :action => 'rpc
  

Далі JS, також тут -> https://gist.github.com/ahoward/7320900

Переглядаючи це, ви можете побачити, що використання бекенд-дій RPC з JS так же просте, як


  var rpc = new RPC(url);

  rpc.call('geo_location', params, callback);

Звичайно, внизу виду ми просто створюємо об'єкт rpc для сторінки з відносним URL

<script>
  jQuery(function(){
 
    var rpc = new RPC(<%= raw url_for(:action => :rpc).to_json %>);
    
  
  });
  
</script>

та ми готові до роботи.

Отже, ось воно - простий спосіб утримати ваші допоміжні RPC JS, щоб вони не засмічували ваші контролери та представлення різними стратегіями, і ще один приклад того, як жорстко послідовні інтерфейси роблять абстракцію можливою та простою.
