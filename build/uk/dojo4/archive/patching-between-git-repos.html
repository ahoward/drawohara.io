Хоча слава та безкоштовні автомобілі приємні, я веду блог, щоб вчитися або, як у цьому випадку, щоб допомогти мені пам'ятати речі.

Я працюю над кількома окремими проектами Rails, які мають спільне походження. Часто виправлення помилки або нова функція в одному проекті повинні бути перенесені в (деякі з) інших.

Оскільки всі проекти знаходяться в своїх власних репозиторіях, зміни не можуть бути злиті за допомогою Git. Ні, це робота для патчів. І коли йдеться про патчінг з Git, є два пости про процес, без яких я не можу обійтися.

Під час патчингу три команди Git, які вступають в гру, git format-patch, git apply та трохи загадковий git am.

git patch досить прямолінійний, надайте список комітів, і він створить диф у форматі patch, який потім можна застосувати Git або командою patch. Те, що я завжди повинен шукати, це шорткат для отримання патчу з останнього коміту.

Нешорткатний спосіб - отримати SHA другий з кінця коміту та передати його format-patch. Чому другий з кінця? SHA представляє стан репозиторію після виконання коміту, отже, остаточний SHA - це те, де ми зараз, а другий з кінця - те, де ми були до наших змін.

Ми створюємо патч таким чином:

git log --oneline
975d4dc Auto-generated CHANGELOG.md
ca1216d Bump for validation fix
git format-patch ca1216d
0001-Auto-generated-CHANGELOG.md.patch

(Ви також можете створити кілька патчів, використовуючи SHA далі в історії.)

Однак є шорткат для отримання другого з кінця коміту, який я знайшов давно в цьому пості:

http://stackoverflow.com/questions/9396240/how-do-i-simply-create-a-patch-from-my-latest-git-commit і є:

git format-patch HEAD^
0001-Auto-generated-CHANGELOG.md.patch

HEAD^ є скороченням від HEAD^1, ^1 означає першого батька коміту, в даному випадку коміт перед HEAD, який, звичайно, є другим з кінця комітом.

Тепер, коли ви маєте свій патч, вам потрібно застосувати його. Для цього я звертаюся до порад, знайдених у https://ariejan.net/2009/10/26/how-to-create-and-apply-a-patch-with-git/

Спочатку ви можете використати:

git apply --stat patch-file

щоб побачити, які зміни фактично містяться в патчі. Потім ви можете використати

git apply --check patch-file

щоб побачити, чи він справді спрацює.

Однак, згідно зі статтею Аріеяна, я насправді не використовую git apply для застосування патчу. Чому ні? Атрибуція.

Альтернатива apply - git am, який призначений для застосування патчів з електронної пошти(в). Що робить його відмінним, так це те, що комітер коміту береться з адреси From у патчі. Для патчів, створених git format-patch, це комітер оригінального коду. З git apply ви є комітером. Таким чином, якщо ви використовуєте git am, ви отримуєте запис про те, хто написав код, а з git apply - запис про те, хто застосував патч.

Що призводить нас до опції --signoff. Переданий до git am, він додає Signed-off-by, що містить ім'я особи, яка застосовує патч, до git log.

git am --signoff < patch file

З цим ми отримуємо і те, й те: коміт, який показує, хто написав код і хто застосував його до цього репозиторію.

І, можливо, тепер я зможу це запам'ятати!
