/dangling-rpc-fruit-off-of-your-rails-controllers
опубліковано: 2013-11-05

Під час роботи над будь-яким додатком Rails, крім "Hello World", вам неминуче знадобиться декілька невеликих AJAX-допоміжників для взаємодії з сервером. Це необхідно для виконання клієнтської логіки. Я не маю на увазі повноцінний API, а радше довільні утиліти, такі як ці:

markdown.change(function(){
  generate_html_on_the_server( markdown.val(), function(html){
  
    preview.html(html);
  
  });
});

або

reservation.change(function(){
  check_availability_on_the_server( reservation.val(), function(data){
  
    if(!data['available']){
     
     alert('цей час недоступний - виберіть інший!');
     
    };
  })
});

і так далі.

Ці функції не є частиною публічного API серверу, вони лише співпрацюють між JS і бекендом для забезпечення роботи відображення. У більшості додатків Rails накопичується багато таких функцій, і виникає питання:

«Куди їх покласти?»

У більшості команд буде три або чотири розробники, які називають і організовують ці функції по-різному, забезпечуючи те, що кодова база перетвориться на безладне спагеті-код за короткий час.

@dojo4 ми абстрагували це за допомогою невеликого шаблону проектування RPC: спочатку ми створюємо невелику мову контролера DSL, яка включається до нашого ApplicationController і дозволяє декларативно визначати допоміжники RPC для кожного контролера.

class ApplicationController < ActionController::Base
  include(RPC)
end

Дивіться реалізацію тут: https://gist.github.com/ahoward/7320900

Ця мова DSL просто визначає одну дію на контролері, яка мультиплексує, який метод використовувати на основі параметрів, і легкий спосіб їх визначення. Вона очікує, що всі дії RPC повернуть хеш і повернуть JSON.

Її реалізація зводиться до:

  def rpc
    which = params['method']
    
    action = @rpc[which]
    
    result = action.call(params)
    
    render :json => result.to_json
  end

Її використання має бути очевидним з коду

class GeoLocationController < ApplicationController
  rpc(:geo_location) do |params|
    geo_location = GeoLocation.geo_locate( params['address'] )
    geo_location.attributes
  end
  
  rpc(:lat_lng) do |params|
   geo_location = GeoLocation.geo_locate( params['address'] )
   { 'lat' => geo_location.lat, 'lng' => geo_location.lng }
  end
end

Використання з JS вимагає двох речей: маршруту та невеликого класа JS. Спочатку маршрут:

  match ':controller#rpc', :action => 'rpc'
  

Далі JS, також тут -> https://gist.github.com/ahoward/7320900

Переглядаючи це, можна побачити, що використання бекенд-дій RPC з JS таке ж просте, як і це:


  var rpc = new RPC(url);

  rpc.call('geo_location', params, callback);

Зазвичай внизу сторінки ми просто створюємо об’єкт RPC для сторінки з відносним URL:

<script>
  jQuery(function(){
 
    var rpc = new RPC(<%= raw url_for(:action => :rpc).to_json %>);
    
  
  });
  
</script>

і ми готові до роботи.

Отже, ось це - простий спосіб утримати ваші допоміжники RPC JS від забруднення контролерів і відображень різними стратегіями та ще один приклад того, як наявність суворо узгоджених інтерфейсів робить абстракцію можливою та простою.
