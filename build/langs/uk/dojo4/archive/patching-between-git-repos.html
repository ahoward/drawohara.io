Хоча слава та безкоштовні автомобілі приємні, я блогую, щоб навчатися, або, як у цьому випадку, щоб допомогти мені пам'ятати речі.

Я працюю над кількома окремими проектами Rails, які мають спільне походження. Часто виправлення помилки або нова функція в одному проекті повинні бути перенесені до (деяких) інших.

Оскільки проекти знаходяться в окремих репозиторіях, зміни не можуть бути злиті за допомогою Git. Ні, це робота для патчів. І коли йдеться про патчінг з Git, існує два пости про процес, без яких я не можу жити.

Під час патчингу, три команди Git, що вступають у гру, це git format-patch, git apply та трохи загадкова git am.

git patch досить простий, надайте список комітів, і він створить диф у форматі патчу, який потім може бути застосований Git або командою patch. Ось що мені завжди доводиться шукати — це шорткат для отримання патча з останнього коміту.

Не шорткат спосіб — це отримати SHA другого останнього коміту та передати його до format-patch. Чому другий останній? SHA представляє стан репозиторію після виконання коміту, отже, остаточний SHA — це те, де ми зараз, а другий останній — те, де ми були до наших змін.

Ми створюємо патч таким чином:

git log --oneline
975d4dc Auto-generated CHANGELOG.md
ca1216d Bump for validation fix
git format-patch ca1216d
0001-Auto-generated-CHANGELOG.md.patch

(Ви також можете створити кілька патчів, використовуючи SHA далі в історії.)

Однак є шорткат для отримання другого останнього коміту, який я знайшов давно в цьому пості:

http://stackoverflow.com/questions/9396240/how-do-i-simply-create-a-patch-from-my-latest-git-commit и є:

git format-patch HEAD^
0001-Auto-generated-CHANGELOG.md.patch

HEAD^ це скорочено від HEAD^1, ^1 означає перший предок коміту, в цьому випадку коміт перед HEAD, який, звичайно, є другим останнім комітом.

Тепер, коли у вас є патч, вам потрібно його застосувати. Для цього я звертаюся до порад, знайдених у https://ariejan.net/2009/10/26/how-to-create-and-apply-a-patch-with-git/

Спочатку ви можете використати:

git apply --stat patch-file

щоб побачити, які зміни фактично містяться в патчі. Потім ви можете використати

git apply --check patch-file

щоб побачити, чи він працюватиме.

Однак, згідно з статтею Ariejan, я насправді не використовую git apply для застосування патча. Чому ні? Атрибуція.

Альтернативою apply є git am, який призначений для застосування патчів з електронної пошти(ок). Чим він відрізняється, так це тим, що автор коміту береться з адреси From у патчі. Для патчів, створених git format-patch це автор оригінального коду. З git apply ви є автором коміту. Отже, якщо ви використовуєте git am, ви отримуєте запис про того, хто написав код, а з git apply — запис про того, хто застосував патч.

Що призводить нас до опції --signoff. Передана до git am вона додає Signed-off-by, що містить ім'я особи, яка застосувала патч у git log.

git am --signoff < patch file

З цим ми отримуємо і те, і інше, коміт, який показує, хто написав код, і хто застосував його до цього репозиторію.

І, можливо, тепер я зможу це запам'ятати!
