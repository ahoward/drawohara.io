Я провів години, досліджуючи та розробляючи, як запобігти запису redis будь-яких значень бази даних на диск, оскільки екземпляр redis буде обробляти конфіденційну інформацію. (У нашому випадку ми зберігаємо cvv протягом 60 хвилин і через відповідність PCI щодо використання кредитних карток, абсолютно не можемо записувати ці значення на диск - ніколи.)

Прочитайте документацію, файл конфігурації, погугліте, протестуйте та навіть спробуйте сказати йому записувати в /dev/null.

Нарешті відповідь прийшла від ‘TheRealBill_here’ на каналі irc #redis:

dbfilename ""

у файлі конфігурації.

Ось посилання на нашу цілу розмову, якщо вас це цікавить.

Я також подав заявку на злиття, щоб додати примітку в файл конфігурації для redis.

Ця відповідь прийшла до мене після того, як я вже прийняв кілька інших рішень щодо конфігурації redis для зберігання інформації про кредитні картки:

  • loglevel має значення за замовчуванням info, яке не друкує кожен читання/запис, тому ми в порядку. Хоча для нашого випадку використання ми, ймовірно, почувалимо себе комфортно, просто повністю вимкнувши ведення журналів.
  • save директива встановлена на save "", щоб гарантовано вимкнути створення знімків.
  • replication не в грі. Під час використання реплікації майстер ініціалізує раба, записуючи свій власний вміст бази даних на диск, надсилаючи рабу цей файл, а раб завантажує дамп бази даних у свою базу даних. Очевидно, ми не можемо цього зробити. (Також зверніть увагу, що якщо ми додамо раба в якийсь момент, він не зможе синхронізуватися з нашою поточною конфігурацією dbfilename "". Дивіться журнал помилки, коли ви викликаєте bgsave з redis-cli самі.)
