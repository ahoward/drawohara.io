Я часто говорив, що завантаження файлів - це самонав'язана DOS-атака. Звичайно, не потрібно далеко шукати приклади, коли, здавалося б, невелика функція додатку викликає хаос на серверах додатків: https://www.google.com/search?q=heroku+image+magick+timed+out

У @dojo4 ми використовуємо можливість Image Magick встановлювати обмеження через довкілля як перший захист, щоб процеси магії були добре поведінкові.

Спочатку трохи підготовки/контексту: ми часто розгортаємо пару файлів RAILS_ROOT/env.yml та RAILS_ROOT/env.rb, які додаток Rails "поглинає" при запуску. Вони не містять жодних залежностей від Rails і можуть використовуватися у фонових скриптах для швидкого розуміння розгорнутої конфігурації Rails без необхідності завантаження всього додатку Rails.

Перший файл - це простий файл YAML із парами ключ=значення.

# файл: RAILS_ROOT/config/env.yml
# RAILS_ENV : production
# RAILS_STAGE : stage
# APP_RUBY : /usr/local/rbenv/versions/1.9.3-p0/bin/ruby
APP_RUBY_VERSION : 1.9.3
# APP_PORT : 4201
# APP_USER : dojo4
# APP_GROUP : dojo4

Другий файл насправді є тим, який вимагається. Він завантажує цей файл і динамічно встановлює подальші змінні оточення:

# файл: RAILS_ROOT/env.rb
#
# спочатку включаємо будь-які налаштування оточення, знайдені в
#
# RAILS_ROOT/env.yml
#
# уважно не перекриваючи жодних вручну встановлених змінних ENV. файл env.yml
# нормально створюється під час розгортання cap.
#
  require 'erb'
  require 'yaml'
  require 'rbconfig'
  
  env_yml = File.expand_path('../env.yml', __FILE__)
  
  if test(?s, env_yml)
    buf = IO.read(env_yml)
    expanded = ERB.new(buf).result(binding)
    config = YAML.load(expanded)
    config.each{|key, val| ENV[key.to_s] ||= val.to_s} if config.is_a?(Hash)
  end

Те, що нас цікавить тут, - це розділ, в якому налаштовується оточення ImageMagick.

Ось він у всій своїй повноті - коментарі повинні надати контекст і пояснення:

# переконайтеся, що RAILS_* встановлені
#
  ENV['RAILS_ENV'] ||= 'development'
  ENV['RAILS_ROOT'] ||= File.dirname(File.dirname(__FILE__))
# встановіть оточення ImageMagick Magick
#
# посилання: http://www.imagemagick.org/script/resources.php
#
# використовуйте систему "convert -list resource" у консолі для перегляду
#
# зрозумійте, що ці налаштування є *на процес* тому багато разів більше кількість
# серверів додатків, які у вас працюють!
#
  tmp = File.join(ENV['RAILS_ROOT'], 'tmp')
# тримайте ваші тимчасові файли поза системним простором, який на AWS є частиною кореневого
# тому!
#
  ENV['TMPDIR']              = tmp
  ENV['MAGICK_TMPDIR']       = tmp
# тримайте лише стільки відкритих дескрипторів файлів одночасно
#
  ENV['MAGICK_FILE_LIMIT']   = 64
# ширина * висота < це значення підходить у пам'ять. інакше він використовує кеш пікселів
#
  ENV['MAGICK_AREA_LIMIT']   = '64MB'
# не їжте більше цієї пам'яті
#
  ENV['MAGICK_MEMORY_LIMIT'] = '256MiB'
# не розміщуйте більше цієї пам'яті
#
  ENV['MAGICK_MAP_LIMIT']    = '1GiB'
# їжте менше цієї кількості диску всього
#
  ENV['MAGICK_DISK_LIMIT']   = '64GiB'
# синхронізуйте записи на диск частіше
#
  ENV['MAGICK_SYNCHRONIZE']  = true
# віддайте процесор частками такої кількості мс
#
  ENV['MAGICK_THROTTLE']     = 256
# 8 хвилин виявиться достатньо довго, щоб змінити розмір проклятого зображення...
#
  ENV['MAGICK_TIME_LIMIT'] = 8 * 60

Додайте смаку на свій смак, тому що тепер ви знаєте, що поведінка ваших процесів зміни розміру зображень знаходиться під вашим контролем, і ви можете використовувати це для захисту від божевільних виділень ресурсів.
