Il futuro dei layout web è promettente, specialmente per lo sviluppatore web esperto le cui tabelle annidate profondamente ancora infestano internet. Ma fino a quando flexbox non sarà pronto per il grande pubblico, i float CSS, il meccanismo di layout principale per framework CSS come Twitter Bootstrap e Zurb Foundation, dovranno bastare.

O forse no?

Alcuni dei brillanti giovani sviluppatori web di oggi potrebbero non essere consapevoli che si è creato un vuoto quando i layout basati su TABELLE sono stati detronizzati. CSS Zen Garden, che ha mostrato le potenzialità dei layout CSS e ha messo un pugnale nel cuore dei layout basati su TABELLE, non ha necessariamente incoronato un successore.

C'era la tecnica del Santo Graal, il Posizionamento Assoluto Fittizio e molte altre fette di PIE. Era un'epoca di sperimentazione, il cui spirito si è un po' affievolito alla luce di tutti i bei framework là fuori (anche se alcune persone sperimentano di tanto in tanto).

Una tecnica che non ha avuto successo è stata quella dei layout CSS a tabelle. Diverse cose hanno cospirato contro di essa:

  1. il titolo del libro che la sosteneva, Tutto quello che sai su CSS è sbagliato, era, ammettiamolo, un po' alienante
  2. il nome stesso della tecnica evocava demoni di tempi passati (layout basati su TABELLE!)
  3. era troppo avanti per il suo tempo, prima che due importanti browser dell'epoca lo supportassero (IE6/IE7)

Recentemente, però, diverse persone hanno rivalutato questa tecnica perché, con poche righe di codice, consente posizionamenti che altre tecniche non possono fare senza hack o JS. Inoltre, offre trucchi di ordine delle fonti che risultano utili nei contesti responsive.

Centratura verticale

Gli sviluppatori frontend sono familiari con la sfida di allineare gli elementi verticalmente in una riga quando le altezze sono sconosciute. Infatti, di solito è il primo problema da risolvere nel HEADER di qualsiasi sito (logo e navigazione).

Le TABELLE rendevano la centratura verticale facile con l'attributo VALIGN. Nei layout basati su float, le cose sono molto più confuse, richiedendo molti trucchi (posizionamento, padding o altezze di linea solitamente).

Entrano in gioco le tabelle CSS, che rendono la centratura verticale banale con la proprietà vertical-align: middle. Per illustrare, prendiamo il seguente markup: 

    <div class="row-table">
      <div class="cell">
        foo
      </div>
      <div class="cell">
        bar
      </div>
      <div class="cell">
        baz
      </div>
      <div class="cell">
        foo<br>bar<br>baz<br>qux
      </div>
    </div>

Combiniamolo con il seguente CSS: 

    .row-table {
      display: table;
      table-layout: fixed;
      width: 100%;
    }

    .cell {
      display: table-cell;
      width: 25%;
      vertical-align: middle;
    }

E otteniamo il seguente layout centrato verticalmente, nonostante il contenuto più alto dell'ultima colonna:

foo
bar
baz
foo
bar
baz
qux

Un ulteriore vantaggio di questo metodo è che la maggior parte degli sviluppatori backend capisce le TABELLE (ti sto guardando, Ara). Quindi, se scrivi stili per altri da usare, puoi creare un sistema di griglia OOCSS con classi e sottoclassi intuitive (".col", ".col-valign-middle", ".col-valign-top", ecc.) per rendere più facile la loro vita.

Colonne di uguale altezza

Con lo stesso markup di cui sopra, è abbastanza facile creare colonne di uguale altezza; aggiungeremo solo alcuni colori di sfondo per dimostrare:

foo
bar
baz
foo
bar
baz
qux

Certo, esistono librerie JS che fanno in modo che le colonne fluttuanti facciano questo, ma ci sono tutte le stranezze di ridimensionamento delle finestre che si devono fare in un contesto responsive per impedire che le altezze delle colonne (calcolate da JS) si riflettano sui viewport più piccoli.

Invece, con una semplice media query, posso far sì che le tabelle CSS si trasformino in blocchi (display: block) o gruppi di tabelle (display: table-header-group) a seconda di ciò che è appropriato per il contesto dato.

Trucchi di ordine delle fonti

Le proprietà delle tabelle CSS possono essere utili quando si desidera rimescolare le carte date dall'ordine delle fonti HTML. Nei layout basati su float, è possibile spingere e tirare gli elementi nei viewport più grandi, ma le cose si impilano nello stesso ordine delle fonti HTML in quelli più piccoli.

Le tabelle CSS consentono di riordinare significativamente le cose, in particolare nei viewport più piccoli. Prendiamo, ad esempio, il nostro esempio precedente:

0
1
2
3

Cambiando l'impostazione di visualizzazione in una delle proprietà dei gruppi di tabelle (display: table-caption, display: table-header-group, display: table-row-group o display: table-footer-group), possiamo impilare gli elementi in modo piuttosto diverso da come appaiono nel layout a riga nei viewport più grandi (rispetto all'ordine delle fonti).

Ad esempio, se volessimo che le colonne sopra si impilassero nel seguente ordine (2,3,1,0), applicheremmo le seguenti proprietà di visualizzazione alle celle (da sinistra a destra): display: table-footer-group (0), display: table-row-group (1), display: table-caption (2), display: table-header-group (3). E le celle si impilerebbero così:

0
1
2
3

Nota: ho dovuto aggiungere una regola width: 100% all'elemento display: table-caption perché altrimenti non avrebbe riempito lo spazio disponibile.

Perché diavolo vorresti farlo?! Beh, potresti fare uno di quei furbi trucchi di navigazione nel footer per i viewport piccoli e vuoi che tutta la navigazione nel tuo HEADER si sposti in fondo alla pagina.

Oppure potresti avere una colonna in una riga che dovrebbe essere visualizzata per prima nei viewport piccoli ma è mostrata a metà di una riga nei viewport più grandi.

Le possibilità NON sono infinite o quasi robuste come quelle di flexbox, ma è qualcosa da tenere nel retro del tuo kit di strumenti.

Note

Le tabelle CSS erano supportate in Internet Explorer a partire da IE8. Altrimenti, il supporto è abbastanza robusto.

Anche se Tutto quello che sai su CSS è sbagliato potrebbe sembrare un titolo di libro scoraggiante, è in realtà una buona lettura, e alcuni dei commenti su Amazon sono divertenti.

Se stai prendendo in considerazione l'idea di mettere qualcosa come una carosello scorrevole all'interno di una cella di una tabella CSS, studia la proprietà table-layout, in particolare il valore "fixed". Questo impedisce alla cella di crescere per contenere il suo contenuto e incidentalmente può ridurre i tempi di rendering perché i browser non devono calcolare tutte le varie larghezze di tutti i vari contenuti delle tue celle.

Ho riscontrato un curioso bug in Firefox riguardo al posizionamento degli elementi all'interno delle celle delle tabelle HTML e CSS. La soluzione "percorso con meno markup", scoperta da un attuale dipendente di Mozilla, è quella di dare sia alla riga che alla cella un'altezza del 100%. Un'altra soluzione ben documentata è quella di utilizzare un elemento all'interno della cella per stabilire un contesto di posizionamento.

Ian Yates ha scritto un eccellente articolo Tuts+ sull'uso delle tabelle CSS per i blocchi pubblicitari, che tendono a rappresentare componenti fissi in un layout fluido.

Ben Frain ha fatto alcuni test di performance molto interessanti confrontando flexbox con le tabelle CSS. I suoi avvertimenti sulle dimensioni del campione e la mancanza di specifiche reali per flexbox dovrebbero essere notati.

Roger Johansson ha notato nell'ottobre 2011 che alcuni lettori di schermo trattano le tabelle CSS come tabelle HTML, cosa che è un problema. Forse è cambiato negli ultimi due anni e mezzo?
