Ogni tanto scrivo di crittografia in Ruby, eppure in qualche modo non sono riuscito a coprire il gem Sekrets di Ara, del Dojo stesso.Sekrets.

Sekrets offre un'interfaccia semplice per creare e gestire file crittografati in Ruby. La sua ragion d'essere è rendere ragionevolmente indolore memorizzare dati sensibili, chiavi API e simili, nei repository Git e quindi accedere a quei dati all'interno della tua app Rails, sia in fase di sviluppo che in produzione. Come tutti i buoni gem, deve solo essere aggiunto al tuo Gemfile:

gem 'sekrets'

E seguito da bundle install. Questo aggiungerà alcune attività rake, inizia con:

rake sekrets:generate:key

Questo crea un file .sekrets.key nella directory principale dell'applicazione. Contiene una password generata casualmente ed è ciò che ci permetterà di decrittografare automaticamente il file di configurazione crittografato. Se lavori con un team, potresti voler sostituire quella password con qualcosa di più facilmente comunicabile, basta farlo prima di eseguire il passaggio successivo.

Vogliamo che questo file esista in fase di sviluppo e che sia disponibile per il nostro deploy, ma metterlo nel repository Git vanificherebbe lo scopo, quindi questo passaggio aggiunge anche .sekrets.key al tuo .gitignore.

Per ripetere, .sekrets.key non dovrebbe essere committato nel tuo repository.

Successivamente esegui:

rake sekrets:generate:config

Questo crea config/sekrets.yml.enc, crittografato con la chiave in .sekrets.key e imposta config/initializers/sekrets.rb.

config/sekrets.yml.enc è un file YAML crittografato. Qui è dove memorizzi tutti i dati che desideri mantenere segreti:

---
api_key: 42
aws:
  access_key_id: XXXXXXXXXXXXXXXXXXXX
  secret_access_key: YYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYY
la_lista_dei_miei_nemici: ['Bob','Kevin','Stuart']
l'inverno_sta_arrivando: true

Puoi modificare il file con:

sekrets edit config/sekrets.yml.enc

O visualizzarlo con:

sekrets read config/sekrets.yml.enc

L'inizializzatore imposta una costante globale SEKRETS. Se il file .sekrets.key esiste, viene popolato con i valori di config/sekrets.yml.enc:

SEKRETS[:aws]
=> {"access_key_id"=>"XXXXXXXXXXXXXXXXXXXX",
    "secret_access_key"=>"YYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYY"}

(SEKRETS è in realtà un Map, una sorta di Hash intelligente, quindi puoi anche ottenere valori tramite il metodo SEKRETS.aws o SEKRETS.l'inverno_sta_arrivando?)

Ora tutti i dati sensibili di cui la tua app ha bisogno sono nel repository e si spostano con essa. Tuttavia, se qualcuno ottiene l'accesso al repository, senza la chiave, non ha comunque accesso ai tuoi segreti. Tutto ciò che devi fare quando aggiungi un nuovo sviluppatore è far loro inserire la chiave nel file .sekrets.key sulla loro macchina locale.

Questo risolve lo sviluppo, e la produzione? Sekrets ti offre due opzioni. In primo luogo, puoi impostare SEKRETS_KEY nell'ambiente del tuo server web, in modo simile a come le persone rendono generalmente disponibili le informazioni sensibili al di fuori della loro base di codice. In alternativa, se usi Capistrano, puoi aggiungere require 'sekrets/capistrano' al tuo Capfile. Questo aggiunge un'attività che copia il .sekrets.key sul server durante il deploy.

Con entrambi i metodi, quando l'App si avvia, Sekrets troverà la sua chiave, decrifterà il file di configurazione e imposterà la costante SEKRETS.

Dovrebbe essere ovvio che i tuoi sekrets, scusa, i segreti sono sicuri solo quanto il tuo server. Qualcuno con accesso al server può recuperare la variabile ambientale SEKRETS_KEY o il file .sekrets.key e usarlo per decrittografare i sekrets.

Tuttavia, questo non è davvero diverso dall'approccio comune di mettere cose sensibili, come le chiavi API, nell'ambiente del server web e accedervi tramite ENV nell'app. In effetti, impostare SEKRETS_KEY nell'ambiente è esattamente lo stesso approccio.

Alla fine, se i dati sensibili sono sul server e in una forma che può essere accessibile dal tuo software, allora sono sicuri solo quanto il server. Sekrets non risolve questo problema. Ciò che fa è mettere quei dati sotto il controllo delle revisioni in modo sicuro e semplificare la condivisione di quei dati tra gli sviluppatori. E questo è un grande vantaggio.
