Como todos los desarrolladores, recibimos muchas solicitudes para crear una aplicación tanto para iOS como para Android. Quieren encontrar una manera de ahorrar tiempo y dinero sin duplicar completamente el trabajo en su aplicación para cada sistema operativo. Por lo tanto, naturalmente hablamos con estos posibles clientes sobre las opciones de arquitectura utilizando contenido HTML y web views. (Nuestro cliente más reciente quería compartir funcionalidades entre una aplicación de iOS y un sitio web, por lo que nos dirigieron especialmente a usar HTML en la aplicación móvil). A lo largo de este camino, siempre surge la idea de usar Phonegap. Hemos recorrido este camino varias veces, y estas son nuestras conclusiones.

Phonegap apesta. Aquí están las razones:

  • Aprender un framework completamente nuevo cuando deberías estar aprendiendo Obj-C o Java.

  • Depender de ellos para soportar las características del SO (¡cruza los dedos para que exista y funcione! ¡cruza los dedos para que soporten cosas nuevas rápidamente!)

  • Desorden organizativo—¿Phonegap? ¿Cordova? ¿Cuál es? (Solo usa Phonegap y estarás bien—Okay… oh, espera, ¡tengo que usar la CLI de cordova para agregar plugins?!)

  • Documentación desactualizada, incorrecta y confusa (en parte debido al punto anterior)

  • La comunidad alrededor está más preocupada por soportar todos los dispositivos del planeta en lugar de ayudar a un desarrollador a tomar algo de contenido HTML y publicarlo sin esfuerzo en uno de los 2 principales SO. (Bueno, eso es para lo que está Phonegap Cloud—intenté usarlo 3 veces, está roto.)

En resumen, Phonegap se centra en hacerte sentir 10% satisfecho en el 100% de los dispositivos del mundo. En nuestra experiencia, a los clientes les importa estar 100% satisfechos solo en 2 de los dispositivos del mundo: iOS y Android.

Entonces, ¿cuál es nuestra solución? ¡Abracemos Obj-C y Java! El contenido HTML en una aplicación es una excelente manera para que las aplicaciones simples tengan una sola base de código en múltiples plataformas, pero necesitas un control detallado sobre toda la aplicación y debes seguir las instrucciones de Apple y Google sobre cómo aprovechar las características de sus SO. Por lo tanto, tenemos frameworks (iOS casi listos para ser open source y Android aún en desarrollo) que contienen una web view y luego usan pequeñas bibliotecas de código abierto para crear un puente entre esa web view y la aplicación nativa. Esto nos permite escribir nuestra propia API de Javascript para llamar a cualquier código nativo que escribamos. (Para iOS, usamos el increíble WebViewJavascriptBridge de Marcus Westin)

Por ejemplo, cuando tenemos contenido HTML que quiere compartir un enlace en Facebook o Twitter, escribimos nuestra propia API de Javascript para decir:

bridge.callHandler('facebookShare', 'http://ducksarethebest.com');
y luego en Objective-C, escribimos el manejador para tomar esa URL y usar el SDK oficial de Facebook para iOS para compartir ese enlace:

[_bridge registerHandler:@"facebook_share" handler:^(id data, WVJBResponseCallback responseCallback) {
FBAppCall *appCall = [FBDialogs presentShareDialogWithLink:[NSURL URLWithString:data] handler:fbHandler];
}];
Esta es una experiencia dramáticamente mejor que dejar que la web view comparta ese enlace porque si el usuario ya tiene la aplicación de Facebook y ha iniciado sesión en ella, el SDK de Facebook no le pedirá que vuelva a iniciar sesión, mientras que el método de compartir de la web view sí lo hará.

Es aún mejor para soluciones donde no hay una alternativa web, como las compras dentro de la aplicación. Y especialmente algo tan vital para el éxito de tu aplicación—los usuarios pagándote dinero—quieres asegurarte de que se haga correctamente de la manera más actualizada y recomendada. La única manera de hacerlo es seguir las directrices de Apple o Google en su SDK oficial.

Este enfoque nos da la flexibilidad de escribir cualquier API de JS que funcione mejor para nosotros, la inmediatez de aprovechar cualquier característica disponible en los SDK de iOS o Android, y mejora nuestras habilidades de desarrollo en un lenguaje y SDK que realmente importan para nuestras carreras a largo plazo. Lo más importante es que el cliente está contento de que tengamos el control total de la aplicación, no dependemos de algún grupo externo para soportar características que Apple o Google lanzaron hace meses, y la aplicación puede ser tan nativa o basada en web como sea necesario para crear la mejor experiencia de usuario.
