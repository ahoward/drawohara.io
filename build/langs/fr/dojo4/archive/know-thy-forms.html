le but était de voir comment les gens réfléchissent aux interfaces HTML.

voici quelques exemples de réponses, regroupées par catégorie, avec des commentaires sur chacune d'elles

fubar doctype

  • Pas de doctype. Cela rend difficile de déterminer la version de HTML utilisée, ce qui pourrait conduire à des réponses intéressantes…
  • Mélange les balises XHTML vides avec les balises HTML.

la plupart des gens ont raté le fait que de nombreuses erreurs dépendaient de la déclaration du doctype.
la manière dont vous utilisez les balises dépend entièrement du doctype.

method=delete

  • ligne 1 : DELETE n'est pas une action supportée dans les formulaires HTML :
  • method=delete n'est pas directement supporté dans la plupart/tous les navigateurs.
  • les formulaires ne supportent pas delete dans les anciens navigateurs
  • Méthode delete
  • form method='DELETE' – ce n'est pas autorisé

bien que nous pensions aux verbes HTTP avec les API client, y compris JavaScript.
pour la majorité des navigateurs, vous obtenez GET et POST. ceci est important si vous avez
envisagé une API basée sur XHTML.

enctype=bunk

  • enctype="multipart/form-data" et non enctype="multipart/file-upload"
  • enc-type="multipart/file-upload n'a pas de sens si notre intention est de DELETE

les bugs d'enctype font toujours perdre une journée. si vous vous trompez, vous perdrez la journée
à poster le nom de votre fichier sous forme de chaîne, au lieu du contenu du fichier encodé.
il est également important de noter que l'enctype dicte simplement comment le
formulaire est encodé et est entièrement orthogonal au verbe HTTP utilisé.

input=foo && input=foo

  • balises de texte dans le premier fieldset ont le nom
  • balises "tags" en double. Ou deux balises avec le même nom à nouveau
  • 2 champs de texte nommés "tags" seul le dernier sera utilisé.
  • Deux entrées avec le même nom 'tags'. La deuxième écraserait la valeur de la 1ère.
  • Deux entrées avec le nom 'foo' (non auto-fermées)
  • entrées de texte avec le même nom écraseraient les entrées avec le nom tags
  • assignations en double à input name='tags'
  • assignations en double à input name='vehicle'

beaucoup de malentendus à ce sujet. il est totalement légitime de poster le même
paramètre > 1 fois dans un formulaire. la spécification le supporte et même l'ancien support cgi
supporte la notion de poster plusieurs valeurs sous la même clé. c'est
simplement une convention que de nombreux contrôleurs les placent dans un hachage où 'le dernier gagne'

vous pouvez facilement le prouver.
