Hier, Ryan Cook et moi avons rencontré l'un de ces redoutables problèmes en production que les développeurs expérimentés connaissent et détestent sur le produit lancé en douceur Moshi Moshi Co Wall Space Finder.

Celui-ci était un casse-tête, les modèles ne pouvaient pas être sauvegardés en environnement de préproduction, mais fonctionnaient localement ou en production. Évidemment, cela semblait lié à RAILS_ENV, ou du moins je le pensais…

Tout d'abord, nous avons fait l'évident, nous avons examiné le code localement en mode développement. Rien, cela fonctionnait parfaitement. Aucun problème.

Ensuite, nous avons utilisé l'incroyable

  ~> cap staging db:suck 
  
  
  ### l'inverse est, vous l'avez deviné, db:blow, parce que le développement doit être plus amusant...

pour aspirer la base de données de préproduction distante dans la base de données locale, répliquant ainsi le code,
les données et l'environnement RAILS_ENV via

  ~> RAILS_ENV=production RAILS_STAGE=staging ./bin/rails server

Une note rapide à ce sujet : Dojo4 exécute tous les déploiements de préproduction en RAILS_ENV=production et différencie la préproduction de la production via une autre variable d'environnement (RAILS_STAGE) précisément pour la raison que nous aimons exercer autant que possible tous les comportements de production en préproduction/qa en protégeant sélectivement uniquement les comportements fous comme la facturation des cartes de crédit via RAILS_STAGE.

Donc, soyez assurés que le problème n'était pas dû au fait que nous avions configuré ./config/environments/staging.rb différemment de ./config/environments/production.rb !

Mais toujours, nous ne pouvions pas répliquer. #WTF!?

Finalement, j'ai instrumenté le déploiement de préproduction pour utiliser #dieüberawesomesauce pry-remote et je suis tombé directement dans le code BOOMing sur le nœud de préproduction.

Et le voilà : une contrainte unique était violée dans la base de données. Pourtant aucun
index uniques n'étaient définis dans le modèle, ni ailleurs. Hmmmmm…

En examinant les journaux git, j'ai découvert que, précédemment, un index unique avait été
défini sur le modèle fautif. Problème résolu, pensais-je, un rapide

  ~> rake db:mongoid:remove_indexes

et. Le problème persistait.

D'accord. Temps de lecture de code. 3 minutes plus tard, le problème était découvert. Dans Mongoid 4, la tâche remove_indexes utilise ce code

      # Retourne la liste des index par modèle qui existent dans la base de données mais ne sont pas
      # spécifiés sur les modèles.
      #
      # @example Retourne la liste des index non utilisés.
      # Mongoid::Tasks::Database.undefined_indexes
      #
      # @return Hash{Class => Array(Hash)} La liste des index non définis par modèle.
      def undefined_indexes(models = ::Mongoid.models)
        undefined_by_model = {}

ref: https://github.com/mongoid/mongoid/blob/master/lib/mongoid/tasks/database.rb#L40

mais, dans Mongoid 3.x, que nous utilisons, il n'a pas une telle logique.

ref: https://github.com/mongoid/mongoid/blob/3.1.0-stable/lib/rails/mongoid.rb

Donc, voilà : Mongoid 4 garantit que tous les index, même ceux qui ne sont plus définis dans le code/dépôt, sont supprimés lorsque les index sont supprimés, tandis que Mongoid 3 laissera ces index dans la base de données !

J'ai décidé d'écrire sur cette expérience car :

  • Avec le recul, je l'ai déjà rencontré moi-même. /cc @spikex
  • Cela souligne comment le développement et les opérations doivent converger pour déboguer les problèmes du monde réel : tout n'est pas sans état et ne vit pas dans le dépôt, et tout l'état ne peut pas être répliqué. Parfois, il faut le faire en direct.
  • Quelqu'un rencontrera inévitablement le même problème et, je l'espère, trouvera cet article via la magie de Google.
  • @modetojoy pourrait envisager ma réflexion actuelle, qui est que nous devrions rétroporter le meilleur comportement de Mongoid 4 dans 3.1.0.

Et les gens se demandent pourquoi nous, ingénieurs, ne pouvons pas estimer le temps et les efforts nécessaires pour corriger un simple bug.
