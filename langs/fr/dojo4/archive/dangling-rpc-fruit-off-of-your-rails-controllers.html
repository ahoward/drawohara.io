lorsque vous travaillez sur une application rails autre qu'un "hello world", vous avez inévitablement besoin de petits helpers ajax pour communiquer avec le serveur afin d'effectuer des logiques côté client. je ne parle pas d'une API en soi, mais d'utilitaires ad-hoc comme celui-ci :

markdown.change(function(){
  generate_html_on_the_server( markdown.val(), function(html){
  
    preview.html(html);
  
  });
});

ou

reservation.change(function(){
  check_availibity_on_the_server( reservation.val(), function(data){
  
    if(!data['available']){
     
     alert('ce moment n'est pas disponible - choisissez-en un autre!');
     
    };
  })
});

etc.

ces types de fonctions ne font pas partie d'une API publique de serveur à serveur, ce sont simplement des points de terminaison js/backend coopératifs nécessaires pour faire fonctionner la vue.

la plupart des applications rails accumuleront de nombreuses fonctions de ce type et la question se pose :

« Où les met-on ? »

dans la plupart des équipes, vous aurez trois ou quatre développeurs nommant et organisant ces fonctions différemment, garantissant ainsi que la base de code se transforme en un désordre de spaghetti cowboy en un rien de temps.

@dojo4 nous avons abstrait cela avec un petit modèle de conception rpc : d'abord, nous avons un petit dsl de contrôleur intégré dans notre ApplicationController qui permet des définitions déclaratives d'aides js 'rpc' par contrôleur.

class ApplicationController < ActionController::Base
  include(RPC)
end

consultez l'implémentation ici : https://gist.github.com/ahoward/7320900

en anglais simple, ce dsl définit une seule action sur le contrôleur qui multiplexe la méthode à employer en fonction des paramètres, et un moyen facile de les définir. il s'attend à ce que toutes les actions rpc renvoient un hachage et renvoient du json.

son implémentation se résume à

  def rpc
    which = params['method']
    
    action = @rpc[which]
    
    result = action.call(params)
    
    render :json => result.to_json
  end

son utilisation devrait être évidente à partir du code

class GeoLocationController < ApplicationController
  rpc(:geo_location) do |params|
    geo_location = GeoLocation.geo_locate( params['address'] )
    geo_location.attributes
  end
  
  rpc(:lat_lng) do |params|
   geo_location = GeoLocation.geo_locate( params['address'] )
   { 'lat' => geo_location.lat, 'lng' => geo_location.lng }
  end
end

l'utilisation depuis js nécessite deux choses : un routage et une petite classe js. d'abord le routage :

  match ':controller#rpc', :action => 'rpc'
  

ensuite, le js, également ici -> https://gist.github.com/ahoward/7320900

en lisant cela, vous pouvez voir que l'utilisation des actions rpc backend depuis js est aussi simple que


  var rpc = new RPC(url);

  rpc.call('geo_location', params, callback);

normalement, en bas d'une vue, nous instancions simplement un objet rpc pour la page avec une url relative

<script>
  jQuery(function(){
 
    var rpc = new RPC(<%= raw url_for(:action => :rpc).to_json %>);
    
  
  });
  
</script>

et nous voilà partis.

donc, voilà - un moyen simple d'empêcher vos aides js rpc de polluer vos contrôleurs et vues avec des stratégies variées et un autre exemple montrant comment des interfaces brutalement cohérentes rendent l'abstraction possible et simple.
