Comme tous les studios de développement, nous avons beaucoup de gens qui viennent nous voir pour créer une application pour iOS et Android. Ils veulent trouver un moyen d'économiser du temps et de l'argent en ne dupliquant pas complètement le travail sur leur application pour chaque système d'exploitation. Naturellement, nous parlons avec ces clients potentiels des options d'architecture utilisant du contenu HTML et des web views. (Notre client le plus récent voulait partager des fonctionnalités entre une application iOS et un site web, nous étions donc particulièrement orientés vers l'utilisation de HTML dans l'application mobile.) Sur ce chemin, l'idée d'utiliser Phonegap vient toujours. Nous avons emprunté cette voie plusieurs fois, et voici nos conclusions.

Phonegap est nul. Voici pourquoi :

  • Apprendre un tout nouveau framework alors que vous devriez apprendre Obj-C ou Java.

  • Compter sur eux pour supporter les fonctionnalités du système d'exploitation (croisez les doigts pour que cela existe et fonctionne ! croisez les doigts pour qu'ils supportent rapidement les nouvelles fonctionnalités !)

  • Désordre organisationnel—Phonegap ? Cordova ? Lequel est-ce ??!?!? (Utilisez simplement Phonegap et tout ira bien—D'accord… oh attendez, je dois utiliser le CLI de cordova pour ajouter des plugins ?!)

  • Documentation obsolète, incorrecte et confuse (en partie due au point précédent)

  • La communauté autour est plus préoccupée par la prise en charge de tous les appareils de la planète plutôt que d'aider un développeur à prendre du contenu HTML et à le publier sans effort sur l'un des 2 principaux systèmes d'exploitation. (Eh bien, c'est à cela que sert Phonegap Cloud—essayé de l'utiliser 3 fois, il est cassé.)

En bref, Phonegap est axé sur le fait de vous rendre 10 % satisfait sur 100 % des appareils du monde. Dans notre expérience, les clients se soucient d'être 100 % satisfaits sur seulement 2 des appareils du monde—iOS et Android.

Alors, quelle est notre solution ? Adoptez Obj-C et Java ! Le contenu HTML dans une application est un excellent moyen pour les applications simples d'avoir une base de code unique sur plusieurs plateformes, mais vous avez besoin d'un contrôle précis sur l'ensemble de l'application et vous devez suivre les instructions d'Apple et de Google sur la façon de tirer parti de leurs fonctionnalités de système d'exploitation. Nous avons donc des frameworks (iOS presque prêt pour l'open source et Android encore en développement) qui contiennent une web view et utilisent ensuite de petites bibliothèques open source pour créer un pont entre cette web view et l'application native. Cela nous permet d'écrire notre propre API JavaScript pour appeler tout code natif que nous écrivons. (Pour iOS, nous utilisons l'incroyable WebViewJavascriptBridge de Marcus Westin)

Par exemple, lorsque nous avons du contenu HTML qui souhaite partager un lien sur Facebook ou Twitter, nous avons écrit notre propre API JavaScript pour dire :

bridge.callHandler('facebookShare', 'http://ducksarethebest.com');
et ensuite en Objective-C, nous avons écrit le gestionnaire pour prendre cette URL et utiliser le SDK officiel Facebook iOS pour partager ce lien :

[_bridge registerHandler:@"facebook_share" handler:^(id data, WVJBResponseCallback responseCallback) {
FBAppCall *appCall = [FBDialogs presentShareDialogWithLink:[NSURL URLWithString:data] handler:fbHandler];
}];
C'est une expérience nettement meilleure que de laisser la web view partager ce lien car si l'utilisateur a déjà l'application Facebook et est connecté, le SDK Facebook ne lui demandera pas de se reconnecter, tandis que la méthode de partage de la web view le fera.

C'est encore mieux pour les solutions où il n'y a pas d'alternative web, comme les achats intégrés. Et surtout quelque chose de si vital pour le succès de votre application—les utilisateurs vous paient de l'argent—vous voulez vous assurer que c'est fait correctement de la manière la plus à jour et recommandée. La seule façon de le faire est de suivre les directives d'Apple ou de Google dans leur SDK officiel.

Cette approche nous donne la flexibilité d'écrire n'importe quelle API JS qui fonctionne le mieux pour nous, l'immédiateté de tirer parti de toute fonctionnalité disponible dans les SDK iOS ou Android, et elle améliore nos compétences en développement dans un langage et un SDK qui compte vraiment pour notre carrière à long terme. Plus important encore, le client est heureux que nous ayons un contrôle total sur l'application, nous ne comptons pas sur un groupe tiers pour supporter des fonctionnalités qu'Apple ou Google ont publiées il y a des mois, et l'application peut être aussi native ou basée sur le web qu'il faut pour créer la meilleure expérience utilisateur.
