Hemos estado utilizando Can.js últimamente para algunos trabajos móviles en tablets. Es una biblioteca ligera de frontend JS para construir aplicaciones web y la hemos estado usando en Phonegap para crear interfaces nativas. El esfuerzo para aprender y aplicar el framework parece haber sido exitoso, así que queremos compartir nuestra experiencia.

Pros

Can.js tiene muchas cualidades que los desarrolladores han llegado a conocer y amar a lo largo de los años. Para listarlas:

  • Ligera – El framework no impone nada. De hecho, parece haber muy poca convención expresada por los desarrolladores.
  • Concisa – Usar objetos Can es un asunto rápido y limpio y tienden a proporcionar interfaces declarativas agradables para cosas como el enlace de eventos y el enrutamiento.
  • Rápida – Can.js se promociona como rápida. No ejecutamos nuestros propios benchmarks, pero no hemos notado nada en contra.
  • Flexible – Dado que el framework no expresa ni implica muchas convenciones, la organización y el uso del código dependen mucho de nosotros, los desarrolladores.
  • Componible – Can viene con un objeto base útil llamado Can.Construct que admite cosas como propiedades estáticas y de prototipo, constructores e herencia. Esto es útil para definir tus propios objetos de aplicación o específicos del dominio de manera consistente.

Molestias

Estuve tentado a llamar a esta lista una lista de “contras”, pero no parecía apropiado ya que son más molestias que contras.

  • Documentación – Parece exhaustiva a primera vista, pero encontré que el texto y los ejemplos no son tan útiles como podrían ser. Creo que un aumento en la diversidad de los ejemplos sería una victoria inmediata.
  • No hay estructura sugerida – Esto también se mencionó anteriormente como una ventaja. Invertimos una cantidad significativa de tiempo creando lo que parecía código de plantilla para comenzar una aplicación rica. No creo que esto pertenezca al framework en sí, pero ejemplos de diferentes metodologías podrían ayudar.
  • Falta de abstracciones de mayor nivel – La mayoría de los objetos Can proporcionados son bastante de bajo nivel desde el punto de vista de la aplicación. Gran parte de la inversión mencionada en el punto anterior se utilizó para crear algunos objetos base útiles para diferentes propósitos para unir Can de diferentes maneras. Esto incluía cosas como una clase de aplicación para adjuntar a window y representar/contener algunos conceptos con estado. También, creamos un objeto base para representar controladores a nivel de página en lugar de controladores a nivel de “widget”.

Conclusión

Recomendamos encarecidamente probar Can.js. Tiene mucho más que no se cubre aquí, como cómo ayuda a prevenir fugas de memoria. No dudes en enviar un correo electrónico a ryan@dojo4.com con cualquier pregunta que tengas.
