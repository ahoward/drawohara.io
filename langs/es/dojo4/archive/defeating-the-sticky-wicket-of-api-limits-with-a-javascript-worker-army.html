TL;DR

cuando tienes demasiado trabajo que hacer en el servidor con respecto a los límites de api

entrega algún trabajo a los navegadores de tus clientes para que lo realicen, enviando los resultados de vuelta a tu base de datos

el problema

la semana pasada estaba tratando con (g-provider) límites de api y demasiadas solicitudes de geolocalización provenientes de un servidor(s). trabajé en eso un poco, haciendo cosas en segundo plano, con reintentos periódicos - ya sabes: lo estándar...

la api en cuestión está limitada por ip, y la documentación recomienda aprovechar esto realizando la mayoría de las llamadas a la api desde el cliente - en js.

ahora, en este caso particular, absolutamente necesitaba que los resultados de las llamadas a la api residieran en el servidor, así que me encontré con esta solución comprometedora (en más de un sentido)

la solución

supongamos que ciertos clientes están dispuestos a consumir su cpu para poblar los datos que les interesa trabajar. en otras palabras, haz que algunos de tus clientes ejecuten trabajos desde una cola de trabajos de javascript para preparar la bomba que están interesados en beber.

en mi caso, tengo ciertas interfaces seleccionadas (las interesadas en la salida de dichos trabajos) ejecutando un pequeño iframe

el js necesario para interactuar con la cola de trabajos es corto y sencillo

todo lo que hace es obtener un trabajo, ejecutarlo y enviarlo de vuelta al servidor, respetando tanto la tasa de procesamiento como el máximo de trabajos para ejecutar.
