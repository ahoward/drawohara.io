Хоча слава та безкоштовні автомобілі приємні, я веду блог, щоб вчитися або, як у цьому випадку, щоб допомогти мені пам'ятати речі.

Я працюю над кількома різними проектами Rails, які мають спільне походження. Часто виправлення помилок або нова функція в одному проекті повинні бути перенесені до (деяких з) інших.

Оскільки проекти живуть у своїх власних репозиторіях, зміни не можуть бути злиті за допомогою Git. Ні, це робота для патчів. І коли йдеться про створення патчів з Git, є два пости про процес, без яких я не можу жити.

При створенні патчів три команди Git, які беруть участь, git format-patch, git apply, і трохи загадковий git am.

git patch досить простий, надайте список комітів, і він створить diff у форматі патча, який потім можна застосувати Git або командою patch. Те, що мені завжди доводиться шукати, це шорткат, щоб отримати патч з останнього коміту.

Не шорткат-спосіб полягає в тому, щоб отримати SHA другого останнього коміту та передати його format-patch. Чому другий останній? SHA представляє стан репозиторію після здійснення коміту, отже, остаточний SHA — це те, де ми зараз, а другий останній — це те, де ми були до наших змін.

Ми генеруємо патч таким чином:

git log --oneline
975d4dc Auto-generated CHANGELOG.md
ca1216d Bump for validation fix
git format-patch ca1216d
0001-Auto-generated-CHANGELOG.md.patch

(Ви також можете створити кілька патчів, використовуючи SHA далі в історії.)

Однак є шорткат для отримання другого останнього коміту, який я знайшов давно в цьому пості:

http://stackoverflow.com/questions/9396240/how-do-i-simply-create-a-patch-from-my-latest-git-commit і є:

git format-patch HEAD^
0001-Auto-generated-CHANGELOG.md.patch

HEAD^ є скороченням від HEAD^1, ^1 означає першого батька коміту, в даному випадку коміт перед HEAD, який, звичайно, є другим останнім комітом.

Тепер, коли у вас є патч, вам потрібно застосувати його. Для цього я звертаюся до поради, знайденої в https://ariejan.net/2009/10/26/how-to-create-and-apply-a-patch-with-git/

Спочатку ви можете використати:

git apply --stat patch-file

щоб побачити, які зміни фактично містяться в патчі. Потім ви можете використати

git apply --check patch-file

щоб побачити, чи він справді спрацює.

Однак, згідно з статтею Аріжана, я насправді не використовую git apply для застосування патча. Чому ні? Атрибуція.

Альтернативою apply є git am, який призначений для застосування патчів з електронної пошти(електронних листів). Що робить його іншим, так це те, що комітер у коміті береться з адреси From у патчі. Для патчів, створених за допомогою git format-patch, це комітер оригінального коду. З git apply ви є комітером. Отже, якщо ви використовуєте git am, ви отримуєте запис про того, хто написав код, а з git apply запис про того, хто застосував патч.

Що приводить нас до опції --signoff. Переданий до git am, він додає Signed-off-by, що містить ім'я особи, яка застосувала патч, до git log.

git am --signoff < patch file

З цим ми отримуємо і те, і інше, коміт, який показує, хто написав код, і хто застосував його до цього репозиторію.

І, можливо, тепер я зможу це запам'ятати!
