je pense que beaucoup de développeurs considèrent le travail de développement comme étant celui de développer des logiciels.

bien sûr, il s'agit de cela !

cependant, il s'agit aussi de développer le développeur.

devenir plus fort.

voici un simple algorithme que l'on peut utiliser pour devenir un développeur plus fort :

  • comprendre complètement le problème

  • concevoir une abstraction

  • implémenter l'abstraction

  • publier l'abstraction

  • solliciter des retours

  • utiliser les retours pour construire la compréhension

c'est assez abstrait pour que certaines personnes comprennent, alors laissez-moi donner un exemple concret contrastant ce développeur

avec celui-ci

dans le développement d'une application qui doit créer des triangles isocèles et droits.

le développeur un utilise le processus suivant :

  • prendre la carte 'isocèle' du tableau. ne pas se donner la peine de lire celle 'triangle droit'.

  • obtenir des échantillons de triangles isocèles. écrire du code qui peut produire de manière fiable uniquement ces types de triangles isocèles que le code attend. puisque faire plus est YAGNI, vérifier, s'assurer que l'intégration continue est verte. et rentrer à la maison.

le développeur deux fait plutôt :

  • passer en revue le tableau. lire les derniers commits. comprendre la feuille de route à 10 000 pieds le développeur fusionne les cartes 'isocèle' et 'droit' en une seule nouvelle carte appelée 'construire un modèle de domaine de triangle' (comprendre le problème)

  • le développeur travaille sur papier, arrivant finalement à (l'abstraction)

  • le développeur implémente du code qui modélise les triangles génériques en utilisant le théorème et les spécialisations pour chacun des types spécifiques 'isocèle' et 'droit' (implémenter l'abstraction)

  • le développeur isole le code de domaine du code non-domaine et pousse une bibliothèque sur github. (publier l'abstraction et solliciter des retours)

  • le développeur fusionne la demande de tirage pour le bug 'division par zéro' plusieurs jours plus tard (utiliser les retours pour approfondir la compréhension)

à la fin, le développeur un coche une case et renforce le processus de son équipe. le développeur deux renforce le code, l'équipe et, surtout, lui-même.

en répétant le processus encore et encore, le développeur un peut se transformer en développeur deux en travaillant de manière répétée pour renforcer les muscles techniques et de domaine. inversement, le développeur deux peut arrêter de s'entraîner et devenir mou comme le développeur un.

la capacité du développeur est comme un muscle : il faut l'utiliser ou le perdre.
