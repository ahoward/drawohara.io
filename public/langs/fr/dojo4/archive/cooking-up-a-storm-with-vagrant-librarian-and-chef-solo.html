Il est assez facile de configurer Apache sur un seul système. Cela devient un peu plus intéressant si vous avez plusieurs vhosts dans la configuration. Cela devient encore plus amusant lorsque vous avez une configuration complexe qui inclut le proxy, la réécriture d'URL et la substitution de texte.

Mais toujours, rien qu'un administrateur système expérimenté ne puisse gérer avec un peu de soin.

Jusqu'à ce que, eh bien, vous deviez le refaire.

C'est là que des outils comme Chef et Puppet entrent en jeu. Les deux outils vous permettent de déclarer et de définir à quoi doivent ressembler les systèmes et comment ils doivent être configurés. Les deux sont puissants et les deux sont activement utilisés pour configurer tout, des postes de travail des développeurs aux grands déploiements de centaines de milliers de machines.

Comme pour le logiciel, vous devez toujours construire et tester la configuration que vous construisez. Vos options pour le faire vont de très coûteux (acheter un ou plusieurs serveurs) à modérément coûteux (exécuter quelques instances sur EC2) à très, très bon marché. L'option très, très bon marché serait de construire et de tester vos recettes sur une ou des machines virtuelles fonctionnant sur votre système. C'est là que Vagrant entre en jeu.

Vagrant est un wrapper autour de la plateforme de virtualisation de bureau VirtualBox d'Oracle. La configuration de Vagrant tourne autour de quelques concepts simples. Vous avez un fichier de configuration nommé Vagrantfile à la racine du projet. Ce fichier contient une configuration de base telle que la boîte de base à utiliser pour le projet et les ports à exposer au système hôte. Et tout aussi important, il peut décrire comment approvisionner la machine virtuelle pour qu'elle réponde aux besoins de votre projet. Bien que Vagrant supporte à la fois Puppet et Chef, j'ai choisi d'utiliser Chef-Solo pour approvisionner ce projet.

Commencer avec Vagrant est simple. La première étape consiste simplement à installer le gem vagrant. Comme tout autre projet Ruby, j'ai choisi d'installer vagrant et quelques autres gems via Bundler. Voici le Gemfile résultant :

source 'https://rubygems.org'
gem 'vagrant'
gem 'librarian'

L'exécution de bundle install a installé les gems nécessaires dans le répertoire du projet. Un rapide rbenv rehash a rendu l'exécutable vagrant disponible pour mon shell.

L'étape suivante consistait à configurer la configuration de base de Vagrant. J'ai exécuté vagrant init puis modifié le Vagrantfile résultant pour qu'il ressemble à ceci :

Vagrant::Config.run do |config|
  # Chaque environnement virtuel Vagrant nécessite une boîte pour se construire.
  config.vm.box = "ubuntu-precise-64"
  
  # L'url d'où la boîte 'config.vm.box' sera récupérée si elle
  # n'existe pas déjà sur le système de l'utilisateur.
  config.vm.box_url = "http://files.vagrantup.com/precise64.box"
end

Cela réalise 2 choses. Premièrement, cela déclare le nom de la boîte sur laquelle nous allons construire et où trouver cette boîte. Un rapide vagrant up téléchargera initialement la boîte de base, l'importera dans le projet et démarrera la VM. Cela prend un certain temps pour télécharger la boîte, mais elle est mise en cache sur votre système pour que vous n'ayez pas besoin de la télécharger à nouveau. Vous pouvez également réutiliser la même boîte de base si vous choisissez de le faire sur d'autres projets. L'appel de vagrant halt arrête la VM et l'appel de vagrant destroy, eh bien, détruit la VM elle-même.

Nous n'avons toujours pas fait grand-chose pour approvisionner notre VM cependant. L'étape suivante consiste à dire à Vagrant comment le faire.

Librarian est un autre gem sympa dans la lignée de Bundler et CocoaPods qui vous permet de déclarer quelles recettes Chef vous utilisez. Dans notre cas, une copie réduite du Cheffile du projet ressemble à ceci :

#!/usr/bin/env ruby
#^syntax detection
site 'http://community.opscode.com/api/v1'
cookbook 'apt'
cookbook 'apache2', '>= 1.0.0'
cookbook 'rbenv', :git => 'git://github.com/fnichol/chef-rbenv.git', :ref => 'v0.7.2'
cookbook 'ruby_build'
cookbook 'openssh', :git => 'git://github.com/fnichol/chef-openssh.git'
cookbook 'git'
cookbook 'build-essential'
cookbook 'vim'
cookbook 'user'
cookbook 'sudo'

Le plus grand avantage est que vous n'avez pas besoin de vendre les livres de cuisine dans vos projets. L'exécution de librarian-chef install téléchargera les livres de cuisine et les installera sous le répertoire cookbooks. Cela génère également un Cheffile.lock pour documenter quelle version des livres de cuisine a été installée.

Je pouvais alors indiquer à Vagrant d'utiliser Chef-Solo pour approvisionner la boîte en mettant à jour le Vagrantfile :

Vagrant::Config.run do |config|
  # Chaque environnement virtuel Vagrant nécessite une boîte pour se construire.
  config.vm.box = "ubuntu-precise-64"
  
  # L'url d'où la boîte 'config.vm.box' sera récupérée si elle
  # n'existe pas déjà sur le système de l'utilisateur.
  config.vm.box_url = "http://files.vagrantup.com/precise64.box"
  
  # Activer l'approvisionnement avec chef solo, en spécifiant un chemin de livres de cuisine, un chemin de rôles
  # chemin, et chemin de data_bags (tous relatifs à ce Vagrantfile), et en ajoutant
  # quelques recettes et/ou rôles.
  config.vm.provision :chef_solo do |chef|
    chef.cookbooks_path = ["cookbooks", "site-cookbooks"]
    chef.add_recipe "apt"
    chef.add_recipe "build-essential"
    chef.add_recipe "ruby_build"
    chef.add_recipe "rbenv::system"
    chef.add_recipe "rbenv::vagrant"
    chef.add_recipe "git"
    chef.add_recipe "user"
    chef.add_recipe "vim"
    chef.add_recipe "sudo"
    chef.add_recipe "openssh"
    chef.add_recipe "apache2"
    
    chef.json = {
      authorization: {
        sudo: {
          passwordless: true,
          groups: ["admin", "sudo"]
        }
      },
      openssh: {
        permit_root_login: "no",
        password_authentication: "yes"
      },
      rbenv: {
        rubies: [ "1.9.3-p327" ],
        upgrade: true,
        global: "1.9.3-p327",
        gems: {
          "1.9.3-p327" => [
            { name: "bundler" },
            { name: "main" },
            { name: "map" },
            { name: "open4" },
            { name: "multi_json" },
            { name: "net-ssh", version: "~> 2.2.0" },
            { name: "aws-sdk" },
            { name: "chef" },
            { name: "ohai" }
          ]
        }
      },
      apache: {
        default_site_enabled: false
      }
    }
  end
end

J'ai rapidement créé un répertoire site-cookbooks et commencé à construire les recettes nécessaires pour configurer et approvisionner le logiciel requis pour cette partie du projet. J'ai pu rapidement appliquer la configuration mise à jour en utilisant vagrant provision ou vagrant reload (si j'avais besoin d'un redémarrage). J'ai même détruit la VM sur laquelle je travaillais et l'ai réapprovisionnée à partir de zéro pour m'assurer que tout fonctionnait.

La récompense pour cet effort a été d'utiliser rapidement les mêmes recettes et configurations pour configurer les serveurs réels qui supportent ce projet. J'étais confiant dans le résultat ayant réussi à le construire et à le tester localement d'abord, tout comme le code que j'écris. D'où l'expression : infrastructure as code.
