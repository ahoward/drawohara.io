Bien que la célébrité et les voitures gratuites soient agréables, la raison pour laquelle je blogue est d'apprendre, ou, comme dans ce cas, de m'aider à me souvenir des choses.

Je travaille sur un certain nombre de projets Rails distincts qui partagent une ascendance commune. Souvent, une correction de bug ou une nouvelle fonctionnalité dans l'un doit être portée vers (certains) des autres.

Comme les projets vivent tous dans leurs propres dépôts, les modifications ne peuvent pas être fusionnées en utilisant Git. Non, c'est un travail pour les patchs. Et quand il s'agit de corriger avec Git, il y a deux articles sur le processus dont je ne peux pas me passer.

Lors de la correction, trois commandes Git entrent en jeu, git format-patch, git apply, et le quelque peu obscur git am.

git patch est assez simple, donnez une liste de commits, il génère une diff au format patch qui peut ensuite être appliquée par Git ou par la commande patch. La chose que j'ai toujours besoin de rechercher est le raccourci pour obtenir un patch à partir du dernier commit.

Le moyen non-raccourci est d'obtenir le SHA du deuxième dernier commit et de le passer à format-patch. Pourquoi le deuxième dernier ? Le SHA représente l'état du dépôt après que le commit soit fait, donc le SHA final est là où nous sommes maintenant, et le deuxième dernier est là où nous étions avant nos changements.

Nous générons le patch ainsi :

git log --oneline
975d4dc CHANGELOG.md généré automatiquement
ca1216d Augmentation pour la correction de validation
git format-patch ca1216d
0001-Auto-generated-CHANGELOG.md.patch

(Vous pouvez également créer plusieurs patchs en utilisant un SHA plus ancien dans l'historique.)

Cependant, il existe un raccourci pour obtenir le deuxième dernier commit que j'ai trouvé il y a longtemps dans ce billet :

http://stackoverflow.com/questions/9396240/how-do-i-simply-create-a-patch-from-my-latest-git-commit et est:

git format-patch HEAD^
0001-Auto-generated-CHANGELOG.md.patch

HEAD^ est le raccourci pour HEAD^1, le ^1 signifie le premier parent du commit, dans ce cas le commit avant HEAD, qui est, bien sûr, le deuxième dernier commit.

Maintenant que vous avez votre patch, vous devez l'appliquer. Pour cela, je me réfère aux conseils trouvés dans https://ariejan.net/2009/10/26/how-to-create-and-apply-a-patch-with-git/

D'abord, vous pouvez utiliser :

git apply --stat patch-file

pour voir quels changements sont réellement dans le patch. Ensuite, vous pouvez utiliser

git apply --check patch-file

pour voir s'il fonctionnera réellement.

Cependant, selon l'article d'Ariejan, je n'utilise pas réellement git apply pour appliquer le patch. Pourquoi pas ? Attribution.

L'alternative à apply est git am, qui est conçu pour appliquer des patchs à partir d'email(s). Ce qui le rend différent, c'est que le committeur sur le commit est pris à partir de l'adresse From dans le patch. Pour les patchs créés par git format-patch, il s'agit du committeur du code original. Avec git apply, vous êtes le committeur. Ainsi, si vous utilisez git am, vous obtenez un enregistrement de qui a écrit le code et avec git apply un enregistrement de qui a appliqué le patch.

Ce qui nous amène à l'option --signoff. Passée à git am, elle ajoute Signed-off-by contenant le nom de la personne appliquant le patch au git log.

git am --signoff < patch file

Avec cela, nous obtenons les deux, un commit qui montre qui a écrit le code et qui l'a appliqué à ce dépôt.

Et maintenant, peut-être que je pourrai m'en souvenir !
