j'ai souvent dit que les téléchargements de fichiers étaient une attaque DOS auto-infligée. certes, on n'a pas à chercher loin pour trouver des exemples d'une fonctionnalité d'application apparemment petite semant le chaos sur de nombreux serveurs d'applications : https://www.google.com/search?q=heroku+image+magick+timed+out

À @dojo4, nous utilisons la capacité d'Image Magick à définir des limites via l'environnement comme première ligne de défense pour maintenir les processus magick bien comportés.

d'abord un peu de configuration/contexte : nous déployons souvent une paire de fichiers RAILS_ROOT/env.yml et RAILS_ROOT/env.rb que l'application rails aspire au démarrage. ils ne contiennent aucune dépendance rails et peuvent être utilisés dans des scripts en arrière-plan pour comprendre rapidement la configuration rails déployée sans avoir besoin de charger l'application rails entière.

le premier fichier est un simple fichier yaml de paires env k=v.

# fichier: RAILS_ROOT/config/env.yml
# RAILS_ENV : production
# RAILS_STAGE : stage
# APP_RUBY : /usr/local/rbenv/versions/1.9.3-p0/bin/ruby
APP_RUBY_VERSION : 1.9.3
# APP_PORT : 4201
# APP_USER : dojo4
# APP_GROUP : dojo4

le second est en fait le fichier qui est requis. il charge ce fichier et a une tentative dynamique de définir d'autres variables d'environnement

# fichier: RAILS_ROOT/env.rb
#
# d'abord, nous intégrons les paramètres d'environnement trouvés dans
#
# RAILS_ROOT/env.yml
#
# en faisant attention à ne pas écraser les variables ENV définies manuellement. le fichier env.yml est
# normalement créé pendant un déploiement cap.
#
  require 'erb'
  require 'yaml'
  require 'rbconfig'
  
  env_yml = File.expand_path('../env.yml', __FILE__)
  
  if test(?s, env_yml)
    buf = IO.read(env_yml)
    expanded = ERB.new(buf).result(binding)
    config = YAML.load(expanded)
    config.each{|key, val| ENV[key.to_s] ||= val.to_s} if config.is_a?(Hash)
  end
