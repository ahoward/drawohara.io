Je parle occasionnellement de cryptage en Ruby, mais d'une manière ou d'une autre, je n'ai pas encore réussi à couvrir le gem Sekrets d'Ara du Dojo’s own Ara’s Sekrets.

Sekrets fournit une interface simple pour créer et gérer des fichiers cryptés en Ruby. Sa raison d'être est de rendre raisonnablement indolore le stockage de données sensibles, de clés API et autres, dans des dépôts Git, puis d'accéder à ces données dans votre application Rails, à la fois en développement et en production. Comme tous les bons gems, il suffit de l'ajouter à votre Gemfile:

gem 'sekrets'

Et suivez avec bundle install. Cela ajoutera quelques tâches rake, commencez par:

rake sekrets:generate:key

Cela crée un fichier .sekrets.key dans le répertoire racine de l'application. Il contient un mot de passe généré aléatoirement et c'est ce qui nous permettra de déchiffrer automatiquement le fichier de configuration crypté. Si vous travaillez en équipe, vous voudrez peut-être remplacer ce mot de passe par quelque chose de plus facilement communicable, faites-le simplement avant d'exécuter l'étape suivante.

Nous voulons que ce fichier existe en développement, et qu'il soit disponible pour notre déploiement, mais le mettre dans le dépôt Git annulerait le but, donc cette étape ajoute également .sekrets.key à votre .gitignore.

Pour réitérer, .sekrets.key ne doit pas être commis dans votre dépôt.

Ensuite, exécutez:

rake sekrets:generate:config

Cela crée config/sekrets.yml.enc, crypté avec la clé dans .sekrets.key et configure config/initializers/sekrets.rb.

config/sekrets.yml.enc est un fichier YAML crypté. C'est ici que vous stockez toutes les données que vous souhaitez garder secrètes:

---
api_key: 42
aws:
  access_key_id: XXXXXXXXXXXXXXXXXXXX
  secret_access_key: YYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYY
la_liste_de_mes_ennemis: ['Bob','Kevin','Stuart']
lhiver_arrive: true

Vous pouvez éditer le fichier avec:

sekrets edit config/sekrets.yml.enc

Ou le visualiser avec:

sekrets read config/sekrets.yml.enc

L'initialiseur configure une constante globale SEKRETS. Si le fichier .sekrets.key existe, il est alors rempli avec les valeurs de config/sekrets.yml.enc :

SEKRETS[:aws]
=> {"access_key_id"=>"XXXXXXXXXXXXXXXXXXXX",
    "secret_access_key"=>"YYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYY"}

(SEKRETS est en réalité un Map, une sorte de Hash intelligent, donc vous pouvez également obtenir des valeurs par méthode SEKRETS.aws ou SEKRETS.lhiver_arrive?)

Maintenant, toutes les données sensibles dont votre application a besoin sont dans le dépôt et se déplacent avec lui. Cependant, si quelqu'un accède au dépôt, sans la clé, il n'a toujours pas accès à vos secrets. Tout ce que vous avez à faire lorsque vous intégrez un nouveau développeur est de lui faire mettre la clé dans le fichier .sekrets.key sur sa machine locale.

Cela prend en charge le développement, et la production ? Sekrets vous donne deux options. Premièrement, vous pouvez définir SEKRETS_KEY dans l'environnement de votre serveur web, de la même manière que les gens rendent généralement les informations sensibles disponibles en dehors de leur base de code. Alternativement, si vous utilisez Capistrano, vous pouvez ajouter require 'sekrets/capistrano' à votre Capfile. Cela ajoute une tâche qui copie le .sekrets.key sur le serveur lors du déploiement.

Avec l'une ou l'autre des méthodes, lorsque l'application démarre, Sekrets trouvera sa clé, déchiffrera le fichier de configuration et configurera la constante SEKRETS.

Il devrait être évident que vos sekrets, désolé, secrets ne sont sécurisés que si votre serveur est sécurisé. Quelqu'un ayant accès au serveur peut récupérer la variable d'environnement SEKRETS_KEY ou le fichier .sekrets.key et l'utiliser pour déchiffrer les sekrets.

Cependant, ce n'est vraiment pas différent de l'approche courante consistant à placer des éléments sensibles, comme les clés API, dans l'environnement du serveur web et à y accéder via ENV dans l'application. En fait, la définition de SEKRETS_KEY dans l'environnement est exactement la même approche.

En fin de compte, si des données sensibles sont sur le serveur et sous une forme accessible par votre logiciel, elles ne sont alors sécurisées que par le serveur. Sekrets ne résout pas ce problème. Ce qu'il fait, c'est mettre ces données sous contrôle de révision de manière sécurisée et simplifier le partage de ces données entre les développeurs. Et c'est un grand succès.
