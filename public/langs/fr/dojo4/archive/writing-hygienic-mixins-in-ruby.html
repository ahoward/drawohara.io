=begin
il est assez courant, en ruby, que les modules fournissent des fonctionnalités via des mixins.
par exemple:
=end
  module Mixin
    def foo
      42
    end
  end
  class C
    include Mixin
  end
  p C.new.foo #=> 42
=begin
bien que pratique, le contenu des mixins, en particulier les modules, peut facilement fuiter
dans la cible de manière involontaire.
Ici, le module 'Util' se retrouve intégré entre Mixin et C:
=end
  reset!
  module Mixin
    module Util
      def Util.foo
        42
      end
    end
  end
  class C
    include Mixin
    p Util.foo #=> 42
    module Util
      def Util.bar
        42.0
      end
    end
    p Util.bar #=> 42.0
  end
=begin
cela pose un défi pour les auteurs de mixins : comment garder le code
organisé *et* fournir un module qui est sûr à mixer dans n'importe quelle cible.
deux approches principales existent:
1) séparer le mixin de l'espace de noms de premier niveau
=end
  reset!
  module M
    module Mixin
      def foo
        42
      end
    end
    module Util
    end
  end
  class C
    include M::Mixin
  end
  p C.new.foo #=> 42
  p C.const_defined?(:Util) #=> false
=begin
ou
2) utiliser const_missing pour permettre des alias de constantes simples dans un espace de noms privé
=end
  reset!
  module M
    module Namespace
      module Util
        def foo
          42
        end
      end
    end
    def Mixin.const_missing(const)
      begin
        Namespace.const_get(const)
      rescue Object
        raise
      end
    end
    def foo
      Util.foo
    end
  end
  class C
    include M::Mixin
  end
  p C.new.foo #=> 42
  p C.const_defined?(:Util) #=> false
=begin
donc, voilà : veuillez réfléchir soigneusement avant d'utiliser des noms communs
dans vos mixins car ils déversent absolument tous leurs internes dans le
mixé.
=end
BEGIN {
  def reset!
    self.class.send :remove_const, :Mixin rescue false
    self.class.send :remove_const, :C rescue false
  end
}
