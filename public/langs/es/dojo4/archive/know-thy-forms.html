El objetivo de que la gente revisara este formulario HTML totalmente roto era solo para
ver el razonamiento de las personas con esto: la base de las interfaces HTML.

A continuación se presentan algunas respuestas de ejemplo, agrupadas aproximadamente por categoría, y comentarios sobre cada una de ellas.

fubar doctype

  • Sin doctype. Dificulta determinar qué versión de HTML es, lo que debería llevar a respuestas interesantes…
  • Mezcla etiquetas vacías de estilo xhtml con etiquetas HTML.

Seguramente, la mayoría de las personas pasaron por alto que muchos errores dependían de la declaración del doctype.
El estilo de las etiquetas que puedes usar depende completamente del doctype.

method=delete

  • línea 1: DELETE no es una acción soportada en formularios HTML:
  • method=delete no es soportado directamente en la mayoría/todos los navegadores.
  • Los formularios no soportan delete en navegadores antiguos.
  • Método delete.
  • form method='DELETE' – eso no está permitido.

Aunque pensamos en los verbos http con las API del cliente, incluida JavaScript.
Para la mayoría de los navegadores, obtienes GET y POST. Esto es importante si has
considerado una API basada en xhtml.

enctype=bunk

  • enctype="multipart/form-data" no es enctype="multipart/file-upload".
  • enc-type="multipart/file-upload no tiene sentido si nuestra intención es eliminar.

Los errores de enctype siempre desperdician un día. Si lo haces mal, desperdiciarás el día
publicando el nombre de tu archivo como una cadena, en lugar del contenido del archivo codificado.
También es importante señalar que el enctype simplemente dicta cómo se codifica el
formulario y es completamente ortogonal al verbo http utilizado.

input=foo && input=foo

  • Las etiquetas de entrada de texto en el primer fieldset tienen el nombre.
  • Campos duplicados “” tags. O dos tags con el mismo nombre nuevamente.
  • 2 campos de texto nombrados “tags” solo se usará el último.
  • Dos inputs con el mismo nombre ‘tags’. El segundo sobrescribiría el valor del primero.
  • Dos inputs con nombre ‘foo’ (no autocerrados).
  • Las entradas de texto con el mismo nombre sobrescribirían las entradas con el nombre tags.
  • Asignaciones duplicadas a input name='tags'.
  • Asignaciones duplicadas a input name='vehicle'.

Muchas ideas equivocadas sobre esto. Es totalmente legítimo publicar el mismo
parámetro > 1 veces en un formulario. La especificación lo soporta y incluso el soporte antiguo de cgi
soporta la noción de publicar múltiples valores bajo la misma clave. Es
simplemente convención que muchos controladores los colocan en un hash donde 'el último gana'

Puedes demostrarlo fácilmente por ti mismo. Publica esto desde un formulario.

  class Parser < Rack::Multipart::Parser
    def setup_parse
      super
    ensure
      @params = KeyList.new
    end
    class KeyList
      def initialize(*args, &block)
        @params = {}
      end
      
      def [](key)
        @params[key]
      end
      
      def []=(key, value)
        (@params[key] ||= []).push(value)
