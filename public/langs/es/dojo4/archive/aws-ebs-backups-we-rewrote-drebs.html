El otro día lanzamos la versión 0.1.0 de DREBS, nuestro script de Recuperación de Desastres para Almacenamiento en Bloques Elásticos. Lo usamos cada hora en todas nuestras 3 instancias de EC2, realizando copias de seguridad de 6 diferentes volúmenes de EBS. Este script es crucial para nosotros y para la comunidad de AWS en general porque Amazon no respalda oficialmente ningún script o herramienta para proporcionar copias de seguridad. Existen varios productos SaaS que proporcionan esta funcionalidad, pero los scripts de código abierto para hackers como nosotros son pocos y distantes entre sí. La nueva versión reescrita está lista para producción y esperamos que la disfruten tanto como nosotros.

El año pasado, el ex empleado de dojo4 y amigo eterno del dojo Garett Shulman lanzó la primera versión de DREBS. Tenía el simple objetivo de permitir que un administrador defina estrategias de copia de seguridad y tareas previas/posteriores a la copia de seguridad (mysqldump, etc.) para cada volumen de EBS montado en una instancia de EC2. Una simple tarea cron ejecuta DREBS cada hora, lo que recorre cada estrategia de copia de seguridad de la instancia de EBS y, si la estrategia indica que debe realizarse una copia de seguridad, DREBS usa la gema right_aws para hacer una copia de seguridad del volumen y subirlo a la cuenta de AWS como una instantánea de EBS.

La nueva versión de DREBS proporciona la misma funcionalidad pero con algunas mejoras:

  • El estado ahora se almacena en una pequeña base de datos sqlite en lugar de un archivo json.
  • Resistente cuando cambian las estrategias.
  • Resistente cuando se eliminan instantáneas en la consola de AWS.
  • Un shell de DREBS que le permite probar su configuración de manera fácil y rápida.

Como ejemplo de configuración, aquí está DREBS en nuestro crontab en todas nuestras máquinas:

0 * * * * /home/dojo4/git/drebs/bin/drebs execute /usr/local/var/drebs_config.yml &> /usr/local/var/drebs_cron.out

y aquí está nuestro archivo de configuración para una de nuestras instancias de EC2:

---
aws_access_key_id: XXXX
aws_secret_access_key: XXX
region: us-west-1
strategies:
- hours_between: 6
  num_to_keep: 12
  mount_point: /dev/sdh
  pre_snapshot_tasks:
  - pg_dump -U postgres -f /ebs/databases/postgresql/backups/project_name_production.sql project_name_production
  post_snapshot_tasks:
- hours_between: 1
  num_to_keep: 2
  mount_point: /dev/sda1
  pre_snapshot_tasks:
  post_snapshot_tasks:
- hours_between: 24
  num_to_keep: 4
  mount_point: /dev/sda1
- hours_between: 96
  num_to_keep: 4
  mount_point: /dev/sda1
log_path: /usr/local/var/drebs.log
email_on_exception: miles@dojo4.com
email_host: smtp.gmail.com
email_port: 587
email_domain: gmail.com
email_user: xxx
email_password: xxx

Como siempre, tenemos más ideas sobre cómo mejorar este script y, si este script le ayuda tanto como a nosotros, le invitamos a hacer un fork del repositorio y enviarnos una solicitud de extracción:

  • Mejorar la cobertura de pruebas
  • Usar la gema whenever para una configuración más fácil de crontab
  • Intervalos de ejecución arbitrarios (instantáneas cada 5 minutos en lugar de cada hora)
  • Claves de API de AWS y otros valores de configuración desde Datos de Instancia de AWS
  • Agregar configuración de ejemplo de acceso de usuario de AWS

Creemos que DREBS es una gran contribución a la comunidad de AWS que nos ha ayudado a construir una excelente arquitectura de implementación que ha alojado sitios web y productos increíbles a lo largo de los años.
