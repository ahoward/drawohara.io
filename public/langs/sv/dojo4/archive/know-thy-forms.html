Syftet med att ha människor som skummar igenom detta helt söndriga html-formuläret var bara att
se människors tänkande i detta: grunden i html-gränssnitt.

Här nedan följer några exempelsvar, indelade i kategorier, och kommentarer om varje.

fubar doctype

  • Ingen doctype. Det gör det svårt att avgöra vilken HTML-variant det är, vilket bör leda till intressanta svar...
  • Blandar xhtml-stil tomma taggar med HTML-taggar.

För säker är det flesta människor som missade att många fel berodde på doctype-deklarationen.
Vilken stil av taggar du får använda beror helt och hållet på doctypen.

method=delete

  • rad 1: DELETE är inte en åtgärd som stöds i HTML-formulär:
  • method=delete stöds inte direkt i de flesta/någon? webbläsare.
  • formulär stöder inte delete i äldre webbläsare
  • Metod delete
  • formulärmetoden='DELETE' - det är inte tillåtet

fastän vi tänker på http-verb med klient-API:er, inklusive javascript.
för en majoritet av webbläsare får du GET och POST. Det är viktigt om du har
tänkt på en xhtml-baserad api.

enctype=bunk

  • enctype="multipart/form-data" inte enctype="multipart/file-upload"
  • enc-type="multipart/file-upload gör ingen mening om vår avsikt är att RADERA

enctype-fel tar alltid en dag. gör det fel och du kommer att slösa bort dagen
med att posta namnet på din fil som en sträng, istället för den faktiska kodade filens
innehåll. Det är också viktigt att notera att enctype enbart dikterar hur
formuläret kodas och är fullständigt ortogonalt till http-verbet som används.

input=foo && input=foo

  • textinmatningstaggar i den första fieldseten har samma namn
  • dubblerade ""-taggar. Eller två taggar med samma namn igen
  • 2 textfält namngivna "tags" endast det sista kommer att användas.
  • Två inmatningar med samma namn 'tags'. Den andra skulle skriva över värdet för den första.
  • Två inmatningar med namnet 'foo' (ej självstängda)
  • textinmatningar med samma namn skulle ersätta re: inmatningar w/ namnet tags
  • dubblera tilldelningar till inmatningsnamnet='tags'
  • dubblera tilldelningar till inmatningsnamnet='vehicle'

Många missförstånd om detta. Det är helt legitimt att posta samma
parameter > 1 gång i ett formulär. specifikationen stöder det och även antika cgi
stöder tanken på att posta flera värden under samma nyckel. Det är
enbart konvention att många kontrollanter dumpa dessa i en hash där 'sista in vinner'

Du kan bevisa detta lätt för dig själv. posta detta från ett formulär

  <input name='answer' value='40'>
  <input name='answer' value='2'>
      
  

och du kommer att se följande i webbläsarkonsolen

# hack the rack parser to show that http DOES post multiple values
# and that, if we want, we could accumulate them

  class Parser < Rack::Multipart::Parser
    def setup_parse
      super
    ensure
      @params = KeyList.new
    end
    class KeyList
      def initialize(*args, &block)
        @params = {}
      end
      
      def [](key)
        @params[key]
      end
      
      def []=(key, value)
        (@params[key] ||= []).push(value)
      end
      
      def key?(key)
        @params.key?(key)
      end
      
      def to_params_hash()
        self
      end
    end
  end

TL:DR;

Din webbstack kan slänga flera värden på golvet genom att ladda en hash,
men rubys cgi.rb-bibliotek, och http generellt sett gör inte. Detta gäller
inmatningar, kryssrutor, etc - allt med namn kan skicka flera värden.

Särskilt rubys cgi.rb gör inte begränsa postning av flera värden under en nyckel

labels are blargh

  • Ingen för på 'Bar: '-etiketten.
  • olänkad etikett: (etikett för vad?)

Etiketter utan en för är helt legitima. när du omsluter en inmatning med en etikett så är 'för' implicerad som den innehållna inmatningen. dessa är ekvivalenta

<label for='foo'>Foo</label> <input name='foo'>
<label for='foo'>Foo <input name='foo'></label>

med den förra som är mycket enklare att hantera i js ( $label.find('input') )

 $label.find('input')
 

vs.

 
  var name = $label.attr('for');
  
  $input = $('name[' + name ']');
