Att multiplexa vilka bilder som ska skickas till vilka enheter är ett komplicerat problem.

Vissa använder enhetsidentifiering (user-agents) för att avgöra vem som vill ha stora
bilder. Ibland används mediefrågor/skärmbredd som en proxy.

Båda är dåliga val.

Idag kan man enkelt önska sig högupplösta bilder som visas på en retinasurfad platta på lokal wifi. Med ‘normala’ bildförbättringsmetoder
vissa enheter, med bra anslutningar och fantastisk pixel-densitet, kommer att få
lågkvalitativa bilder.

En platta på en långsam anslutning i landskapsorientering kan ha en skärm
med en godtycklig (768px) tröskel och levereras stora bilder
designade för bredbandsanslutna datorer.

Problemet förvärras av att enhets- och/eller skärmbreddsförbättringar enkelt kan
leda till att en liten bild skickas till en enhet med en högupplöst skärm
som orsakar verkliga layoutproblem: en 320px bild kan ta upp hela skärmen på
vissa iphones, medan den på andra kommer att antingen uppta 1/2 av skärmen eller sträckas för att fylla hela 640 och se dåligt ut.

Vad om, i stället, vi enkelt alltid försökte leverera bilder med högsta kvalitet
till alla enheter och erbjöd lägre kvalitetsbilder om vi bestämde att detta var
ett dåligt beslut vid körning?

Metoden för jquery.bires är just detta.

Innehållet kommer först med lågkvalitetsbilder, vilket gör sidladdningstiden
så snabb som möjligt och ger användaren hela innehållet. Efter att alla
lågkvalitetsbilder har laddats börjar jquery.bires ersätta lågkvalitetsbilder
med högkvalitetsbilder seriellt, och mäter tiden det tar att ersätta varje bild. Om någon
ersättning tycks ta för lång tid stoppas förbättringen helt enkelt. Eftersom
jquery.bires laddar de större bilderna en i taget förblir gränssnittet
svarande på användarklick och interaktioner.

Nyckeln här är att, med dagens högupplösta mobilskärmar, är det faktiskt
bandbredd och inte skärmstorlek som bör avgöra vem som får högkvalitetsbilder.

Den här metoden resulterar i att den fina plattan på lokal wifi får den bästa
möjliga upplevelsen samtidigt som sidor som visas på en dator
kopplad via en 3G-anslutning inte dödar en dataplan och laddar mycket
långsamt.

Bäst av allt, istället för att hantera listor över user-agents, gissa om olika
brytpunkter, osv - koden använder enkelt alltid samma strategi att försöka
hårt att göra sidor att ladda snabbt och bli så bra som möjligt senare.

(function(namespace){
  // the hi-res image downloader object
  //
    var bires = {};
    bires.selector      = 'img.bires';
    bires.too_damn_slow = 420;
    bires.debug         = false;
    bires.debug_delay   = 420;
  // downloader class and factory method
  //
    bires.downloader = new Function();
    bires.download = function(){
      var args = Array.prototype.slice.apply(arguments);
      var downloader = new bires.downloader;
      downloader.initialize.apply(downloader, args);
      jQuery(function(){
        jQuery('body').imagesLoaded(
          function(){ downloader.download() }
        );
      });
      return(downloader);
    };
    bires.downloader.prototype.initialize = function(){
      var downloader = this;
      var args = Array.prototype.slice.apply(arguments);
      var options = {};
      var opts = ['selector', 'too_damn_slow', 'debug', 'debug_delay'];
      if(args.length == 1 && typeof(args[0]) == 'object'){
        options = args[0];
      } else {
        for(var i = 0; i < opts.length; i++){
          options[opts[i]] = args[i];
        }
      }
      jQuery.map(
        opts,
        function(o){ downloader[o] = options[o] || bires[o]; }
      );
