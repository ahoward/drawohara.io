Igår Ryan Cook och jag stötte på ett av de fruktade produktionsproblemen som erfarna utvecklare känner och hatar på den mjukt lanserade Moshi Moshi Co produkten Wall Space Finder.

Det här var ett svårt problem, modeller skulle misslyckas med att spara i staging men inte lokalt eller i produktion. Uppenbarligen var det relaterat till RAILS_ENV, eller så trodde jag...

Först gjorde vi det uppenbara, tittade på koden lokalt i utvecklingsläge. Inget, det fungerade totalt. Inga problem.

Sedan använde vi den fantastiska

  ~> cap staging db:suck 
  
  
  ### den inversa är, som du gissade, db:blow, för att utveckling ska vara mer roligt...

för att suga den fjärranslutna stagingsdatabasen till den lokala databasen, och därmed replikera kod,
data och RAILS_ENV via

  ~> RAILS_ENV=production RAILS_STAGE=staging ./bin/rails server

En snabb notering om det: Dojo4 kör alla stagingsdeployment i RAILS_ENV=production och skiljer mellan staging och produktion via en annan miljövariabel (RAILS_STAGE) precis för den anledningen att vi vill öva alla produktionsbeteenden i staging/qa där det är möjligt - selektivt skydda endast galna beteenden som att debitera kreditkort via RAILS_STAGE.

Så du kan vara säker på att problemet inte berodde på att vi hade ./config/environments/staging.rb konfigurerat annorlunda än ./config/environments/production.rb!

Men ändå kunde vi inte replikera. #VAFAN!?

Till slut instrumenterade jag stagingsdeployment för att använda #dieüberawesomesauce pry-remote och hoppade rakt in i den BOOMande koden på stagingsnoden.

Och där var det: ett unikt villkor bröts i databasen. Men inga
unika index definierades i modellen eller någon annanstans. Hrmmmmm…

Genom att granska git-loggarna hittade jag att ett unikt index hadde definierats
på den felande modellen tidigare. Problem löst trodde jag, en snabb

  ~> rake db:mongoid:remove_indexes

och. Problemet kvarstod.

OK. Kodläsningstid. 3 minuter senare upptäcktes problemet. I Mongoid 4 använder remove_indexes tasken den här koden

      # Returnera listan över index per modell som finns i databasen men inte
      # specificeras i modellerna.
      #
      # @example Returnera listan över oanvända index.
      # Mongoid::Tasks::Database.undefined_indexes
      #
      # @return Hash{Class => Array(Hash)} Listan över odefinierade index per modell.
      def undefined_indexes(models = ::Mongoid.models)
        undefined_by_model = {}

ref: https://github.com/mongoid/mongoid/blob/master/lib/mongoid/tasks/database.rb#L40

men, i Mongoid 3.x, som vi använder, har den ingen sådan logik.

ref: https://github.com/mongoid/mongoid/blob/3.1.0-stable/lib/rails/mongoid.rb

Så där har du det: Mongoid 4 säkerställer att alla index, även de som inte längre definieras i koden/repo, tas bort när index tas bort, medan Mongoid 3 kommer att lämna dessa index liggande i databasen!

Jag bestämde mig för att skriva om den här upplevelsen eftersom:

  • 20/20 efterklokskhet jag har stött på det tidigare själv. /cc @spikex
  • Det understryker hur utveckling och dev-ops måste konvergera för att felsöka verkliga problem: inte allt är tillståndslöst och finns i repo, och inte allt tillstånd kan replikeras. Ibland måste du göra det live.
  • Någon kommer säkert att stöta på samma problem och, jag hoppas, hitta den här posten via trollkarlarna på google.
  • @modetojoy kan överväga min nuvarande tankegång, som är att vi bör backporta det bättre beteendet från Mongoid 4 till 3.1.0.

Och folk undrar varför vi ingenjörer inte kan uppskatta tiden och arbetet att åtgärda en enkel bugg.

P.S. Några av er läsare kanske undrar hur jag fixade det. Jag definierade bara indexet i konsolen så att Mongoid kände till det, och använde sedan metoderna på modellnivå för att ta bort det
