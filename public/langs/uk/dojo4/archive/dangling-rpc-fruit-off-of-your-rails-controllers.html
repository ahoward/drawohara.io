При роботі з будь-яким нетривіальним додатком Rails ви неминуче потребуєте невеликих AJAX-допоміжників для спілкування з сервером, щоб виконувати логіку на клієнтській стороні. Я маю на увазі не API як таке, а радше ад-гок утиліти, такі як ці:

markdown.change(function(){
  generate_html_on_the_server( markdown.val(), function(html){
  
    preview.html(html);
  
  });
});

або

reservation.change(function(){
  check_availibity_on_the_server( reservation.val(), function(data){
  
    if(!data['available']){
     
     alert('Цей час недоступний - виберіть інший!');
     
    };
  })
});

і т.д.

Ці типи функцій не є частиною публічного API сервер-сервер, вони просто співпрацюють з JS/backend-ендпоінтами, які необхідні для виконання функцій перегляду.

Більшість додатків Rails накопичать багато таких функцій, і виникає питання:

“Куди їх покласти?”

У більшості команд буде три чи чотири розробники, які називають та організовують ці функції по-різному, забезпечуючи, щоб кодова база перетворилася на безладне спагеті-ковбойське болото в короткі терміни.

У @dojo4 ми абстрагували це за допомогою невеликого шаблону проектування RPC: спочатку у нас є невеликий DSL-контролер, який включено до нашого ApplicationController, що дозволяє декларативні визначення допоміжників RPC JS на основі контролера.

class ApplicationController < ActionController::Base
  include(RPC)
end

Ось реалізація: https://gist.github.com/ahoward/7320900

По-англійськи цей DSL просто визначає одну дію на контролері, яка багаторазово використовує метод на основі параметрів, і простий спосіб визначити їх. Він очікує, що всі дії RPC повертатимуть хеш і віддаватимуть JSON.

Його реалізація зводиться до

  def rpc
    which = params['method']
    
    action = @rpc[which]
    
    result = action.call(params)
    
    render :json => result.to_json
  end

його використання має бути очевидним з коду

    geo_location.attributes
  end
  
  rpc(:lat_lng) do |params|
