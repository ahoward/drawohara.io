Quando ti connetti a un server web in modo sicuro utilizzando HTTPS, la sicurezza viene negoziata utilizzando TLS. Due cose accadono: l'identità del server viene verificata e la connessione viene crittografata.

La verifica è importante: non ha senso che la connessione sia crittografata se in qualche modo sei stato reindirizzato a un server malintenzionato. Tuttavia, questa verifica può essere problematica se un server web serve più di un hostname.

Puoi leggere i dettagli cruenti, ma la versione semplificata del processo è che il server invia un certificato di chiave pubblica firmato che deve corrispondere al nome host nell'URL. Se un client naviga su dojo4.com, il certificato deve essere per dojo4.com; se non lo è, il browser visualizza un avvertimento spaventoso.

Tecnicamente, è possibile avere più hostname su un certificato; in effetti, è comune avere, ad esempio, sia "dojo4.com" che "www.dojo.com" per completezza. Tuttavia, è un fastidio incredibile aggiungere e rimuovere hostname da un certificato. È necessario che l'emittente ne generi uno nuovo e revoci il vecchio. Inoltre, se stai lavorando con una Content Delivery Network, è improbabile che aggiungano i tuoi hostname al loro certificato.

Originariamente, TLS supportava un solo certificato per server web (o più correttamente, per indirizzo IP collegato al server web). Server Name Indication (SNI) è stato aggiunto a TLS per risolvere questo problema. All'inizio della negoziazione TLS, il client comunica al server il nome dell'host a cui sta cercando di connettersi e il server può quindi selezionare e inviare il file di certificato corretto. Problema risolto!

Tranne che... Non tutti i browser supportano SNI. Tutti sanno questo e, di conseguenza, tendono a saltare SNI e passare direttamente agli IP dedicati per sito o addirittura a più server. Questa è un'opzione costosa, specialmente quando si lavora con CDN come CloudFront. Quando mi sono trovato in questa situazione, ho deciso di vedere cosa significava davvero "non tutti i browser".

Si scopre che SNI è ampiamente supportato, con i principali problemi che riguardano IE8 e versioni precedenti e qualsiasi versione di IE in esecuzione su Windows XP (perché la libreria del sistema operativo sottostante non supporta SNI). Ci sono anche alcune vecchie versioni di Android che mancano del supporto.

Quindi, la maggior parte dei visitatori non avrà problemi con SNI e il gruppo che ne ha è abbastanza piccolo da poter essere gestito come caso speciale.

Per i browser senza supporto SNI, la soluzione è reindirizzarli a un certificato che funzionerà o a una pagina sarcastica "aggiorna il tuo browser". Se cerchi su Google, troverai un sacco di soluzioni per creare whitelist di browser buoni e/o blacklist di quelli cattivi e poi utilizzare queste liste nelle regole di reindirizzamento lato server. Brutto. Le liste devono essere mantenute e, a seconda del server, interrompono la cache.

C'è un modo più intelligente. Mentre navigavo in un mare di configurazioni di reindirizzamento Apache di esempio, l'ho trovato in questo post. L'idea principale del post può essere distillata in questo: se un browser che non supporta SNI tenta di caricare contenuti SNI, riceverà un errore. Se testiamo questo in background e differenziamo tra errore e successo, possiamo reindirizzare il visitatore di conseguenza. E il modo più semplice per farlo è provare ad aggiungere un'immagine di un pixel alla pagina.

In codice appare così:

function secure_redirect() {
   var img=document.createElement('img'); // crea un elemento img.
   // Imposta l'attributo src su un URL SNI di un'immagine di un pixel
   img.src='https://www.example.org/pixel.gif';
   // Questo viene eseguito se SNI funziona.
   img.onload = function() {
      // Reindirizza alla pagina sicura.
      window.location.href = "https://example.org/";
   };
   // Questo viene eseguito se SNI non funziona.
   img.onerror = function(e) {
      // Reindirizza altrove
      window.location.href = "http://example.org/snarky-old-browser-message";
   };
   // Non visualizzare effettivamente l'immagine
   img.style.display='none';
   // ma appendila alla pagina in modo che venga caricata.
   document.body.appendChild(img);
  }

Qui sto sfruttando due callback HTML sul tag img, 'onLoad', che viene attivato quando un'immagine termina il caricamento, e 'onError', che viene attivato se l'immagine non può essere caricata. Se un browser non supporta SNI, l'immagine non riuscirà a caricarsi a causa di un errore del certificato, attivando 'onError'. Tuttavia, poiché stiamo aggiungendo l'immagine a una pagina già caricata, non solleverà un errore nel browser.

Ora possiamo testare SNI e gestire la mancanza di supporto in modo elegante. Natale è salvo!

Tuttavia, ciò a cui siamo effettivamente arrivati è qualcosa di più intelligente. Nota che il codice non sta effettivamente testando SNI, ma solo la capacità di caricare l'immagine in modo sicuro. Se l'URL HTTPS in questione non richiede effettivamente SNI, c'è un solo certificato o il primo certificato corrisponde al dominio richiesto, funziona comunque. Il problema è stato ridotto a "Questo visitatore può visualizzare il sito sicuro o no?" e alla fine della giornata, è tutto ciò che ci interessa veramente.
