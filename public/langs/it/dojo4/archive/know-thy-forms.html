L'obiettivo di far scorrere alle persone questo modulo HTML completamente rotto era semplicemente
vedere il filo del pensiero delle persone con questo: la spina dorsale delle interfacce HTML

Seguono alcune risposte di esempio, raggruppate approssimativamente per categoria, e commenti su ciascuna di esse

fubar doctype

  • Nessun doctype. Rende difficile capire quale versione di HTML sia, il che dovrebbe portare a risposte interessanti...
  • Mescola tag vuoti stile xhtml con tag HTML.

Sicuramente la maggior parte delle persone ha perso il fatto che molti errori dipendevano dalla dichiarazione del doctype.
lo stile dei tag che puoi usare dipende interamente dal doctype.

method=delete

  • riga 1: DELETE non è un'azione supportata nei moduli HTML:
  • method=delete non è direttamente supportato nella maggior parte/tutti? i browser.
  • i moduli non supportano delete nei browser più vecchi
  • Metodo delete
  • form method='DELETE' – non è permesso

anche se pensiamo ai verbi http con le api client, incluso javascript.
per la maggior parte dei browser ottieni GET e POST. questo è importante se hai
considerato un'api basata su xhtml.

enctype=bunk

  • enctype="multipart/form-data" non enctype="multipart/file-upload"
  • enc-type="multipart/file-upload non ha senso se la nostra intenzione è DELETE

i bug di enctype fanno sempre perdere un giorno. sbagli e perderai il giorno
pubblicando il nome del tuo file come una stringa, invece del contenuto del file effettivamente codificato
è anche importante notare che l'enctype detta semplicemente come viene codificato il
modulo ed è completamente ortogonale al verbo http utilizzato.

input=foo && input=foo

  • i tag di input di testo nel primo fieldset hanno il nome
  • tag duplicati "". O due tag con lo stesso nome di nuovo
  • 2 campi di testo chiamati "tags" verrà utilizzato solo l'ultimo.
  • Due input con lo stesso nome 'tags'. Il secondo sovrascriverebbe il valore del primo.
  • Due input con nome 'foo' (non autoconclusi)
  • gli input di testo con lo stesso nome sovrascriverebbero re: input con nome tags
  • assegnazioni duplicate a input name='tags'
  • assegnazioni duplicate a input name='vehicle'

molte incomprensioni su questo. è totalmente legittimo pubblicare lo stesso
parametro > 1 volte in un modulo. la specifica lo supporta e anche l'antico supporto cgi
supporta la nozione di pubblicare più valori sotto la stessa chiave. è
semplicemente convenzione che molti controller scaricano questi in un hash dove 'l'ultimo vince'
