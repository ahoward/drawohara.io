I deployment delle applicazioni web sono complessi fin dall'inizio. Spesso si inizia con almeno un deployment di staging e uno di produzione. Aggiungi a questo il multi-tenancy, le configurazioni per le integrazioni di terze parti come l'autenticazione di Facebook o Salesforce, le configurazioni per lo storage cloud e le reti di distribuzione dei contenuti, gli ambienti di integrazione continua, il tema multi-client, le configurazioni SSL, le configurazioni vhost, i processi di build per i frontend js, la configurazione dei task in background, la configurazione dell'elaborazione degli upload, l'elenco potrebbe continuare... esplosione combinatoria.

In dojo4 abbiamo uno stack di applicazioni web basato su rails e utilizziamo Multistage Capistrano per il deployment sui nodi AWS EC2. Abbiamo trovato una strategia semplice per gestire la grande varietà di configurazioni e requisiti di deployment che i nostri clienti ci hanno presentato. La strategia prevede un rake task che utilizza una variabile di ambiente RAILS_STAGE basata sullo stage Multistage di Capistrano. Impostiamo una struttura di file per configurazioni specifiche di deployment in repo_root/config/deploy/files/RAILS_STAGE che è strutturata in modo identico al file system del repository di origine. Durante il deployment cap, prima del collegamento simbolico corrente o dei processi di build, il rake task copia ricorsivamente repo_root/config/deploy/files/RAILS_STAGE in repo_root.

Abbiamo scoperto che questa strategia è molto efficace nel mantenere le configurazioni di deployment complesse organizzate in modo chiaro, facili da comprendere e senza aggiungere molta complessità aggiuntiva a un'architettura già complessa.

Il gist qui sotto dimostra questa idea. Inizialmente, questo task sfruttava varie librerie rails. Tuttavia, sono riuscito a fattorizzare le dipendenze rails mantenendo la funzionalità del task in modo da poterlo utilizzare con qualsiasi stack di applicazioni.
