Anche se la fama e le auto gratuite sono belle, il motivo per cui bloggo è per imparare, o, come in questo caso, per aiutarmi a ricordare le cose.

Lavoro su una serie di progetti Rails distinti che condividono un'origine comune. Spesso una correzione di bug o una nuova funzione in uno deve essere portata sugli altri.

Poiché i progetti vivono tutti nei loro repository, le modifiche non possono essere unite utilizzando Git. No, questo è un lavoro per le patch. E quando si tratta di patch con Git, ci sono due post sul processo di cui non posso fare a meno.

Quando si applica una patch, tre comandi Git entrano in gioco, git format-patch, git apply, e il leggermente oscuro git am.

git patch è piuttosto semplice, dai un elenco di commit, genera una diff in formato patch che può essere applicata da Git o dal comando patch. La cosa che devo sempre cercare è la scorciatoia per ottenere una patch dall'ultimo commit.

Il modo non scorciato è ottenere lo SHA del secondo ultimo commit e passarlo a format-patch. Perché il secondo ultimo? Lo SHA rappresenta lo stato del repo dopo che il commit è stato effettuato, quindi lo SHA finale è dove ci troviamo ora, e il secondo ultimo è dove eravamo prima delle nostre modifiche.

Generiamo la patch come segue:

git log --oneline
975d4dc CHANGELOG.md generato automaticamente
ca1216d Aggiornamento per la correzione della validazione
git format-patch ca1216d
0001-CHANGELOG.md-generato-automaticamente.patch

(È possibile creare più patch utilizzando uno SHA più indietro nella cronologia.)

Tuttavia, c'è una scorciatoia per ottenere il secondo ultimo commit che ho trovato molto tempo fa in questo post:

http://stackoverflow.com/questions/9396240/how-do-i-simply-create-a-patch-from-my-latest-git-commit ed è:

git format-patch HEAD^
0001-CHANGELOG.md-generato-automaticamente.patch

HEAD^ è l'abbreviazione di HEAD^1, il ^1 significa il primo genitore del commit, in questo caso il commit prima di HEAD, che è, naturalmente, il secondo ultimo commit.

Ora che hai la tua patch devi applicarla. Per questo, mi rivolgo al consiglio trovato in https://ariejan.net/2009/10/26/how-to-create-and-apply-a-patch-with-git/

Prima puoi usare:

git apply --stat file-patch

per vedere quali modifiche sono effettivamente nel patch. Poi puoi usare

git apply --check file-patch

per vedere se funzionerà effettivamente.

Tuttavia, secondo l'articolo di Ariejan, non utilizzo effettivamente git apply per applicare la patch. Perché no? Attribuzione.

L'alternativa a apply è git am, che è progettato per applicare patch da email(s). Ciò che lo rende diverso è che il committer del commit viene preso dall'indirizzo From nel patch. Per le patch create da git format-patch questo è il committer del codice originale. Con git apply sei il committer. Pertanto, se usi git am, ottieni un registro di chi ha scritto il codice e con git apply un registro di chi ha applicato la patch.

Che ci porta all'opzione --signoff. Passata a git am aggiunge Signed-off-by contenente il nome della persona che applica la patch al git log.

git am --signoff < file-patch

Con questo otteniamo entrambi, un commit che mostra chi ha scritto il codice e chi lo ha applicato a questo repo.

E ora forse riuscirò a ricordarlo!
